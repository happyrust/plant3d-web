# 模型加载策略升级方案：从 JSON Bundle 到 Parquet

## 1. 背景与目标

当前模型加载采用 "按次生成 JSON Bundle" 的模式（`manifest.json` + `instances/*.json`）。
**存在问题**：
1. **重复计算**：每次请求都重新生成数据，无法有效利用已有的生成结果。
2. **缓存困难**：Bundle 是临时的，难以在多次会话间复用。
3. **并发限制**：生成过程阻塞，无法支持流式持续生成和加载。

**升级目标**：
1. **持久化存储**：按数据库编号（`dbno`）将模型数据保存为 `Parquet` 文件。
2. **增量生成**：每次生成时检测已存在的数据，仅生成新增部分。
3. **高效加载**：前端直接读取 Parquet 文件（列式存储），提高解析和加载效率。

---

## 2. 后端开发方案 (gen-model-fork)

### 2.1 技术选型
*   **Parquet 处理**: 使用 `polars` 或 `parquet` crate 进行高效读写。建议使用 `polars` 以便利用其强大的 DataFrame 操作进行去重和查询。
*   **存储结构**: 基于文件系统的分区存储。

### 2.2 存储目录结构
```text
output/
  └── database_models/
      └── {dbno}/              # 按数据库编号分区
          ├── meta.json        # 元数据（版本、最后更新时间、包含的 refnos 索引）
          ├── part_0001.parquet # 数据分片 1
          ├── part_0002.parquet # 数据分片 2
          └── ...
```

### 2.3 数据 Schema (Parquet Columns)

| 列名 | 类型 | 说明 |
| :--- | :--- | :--- |
| `refno` | String | 唯一标识符 `24383_73962` |
| `noun` | String | 节点类型 `BRAN`, `EQUI` |
| `geo_hash` | String | 几何体哈希 (指向 .glb) |
| `transform` | List<Float> | 16位变换矩阵 [f32; 16] |
| `aabb` | List<Float> | 包围盒 [min_x, min_y, min_z, max_x, max_y, max_z] |
| `is_tubi` | Boolean | 是否为管道直段 |
| `properties` | String (JSON) | 额外的属性数据 (可选) |

### 2.4 生成流程改造

1.  **接收请求**: 收到生成请求 (包含 `refnos`)。
2.  **存在性检查**:
    *   加载目标 `dbno` 下的所有 Parquet 文件的 `refno` 列（Polars lazy load 极快）。
    *   筛选出**未存在**的 `refnos` 进行生成。
    *   如果所有 refnos 都已存在，直接返回成功。
3.  **增量生成**:
    *   对缺失的 refnos 执行原有的几何生成逻辑 (`gen_all_geos_data`)。
4.  **数据写入**:
    *   将新生成的实例数据转换为 DataFrame。
    *   写入一个新的 `part_{timestamp}.parquet` 文件。
    *   *可选：定期执行 Compaction (合并小文件) 任务。*

### 2.5 API 变更

*   `POST /api/model/generate`: 仅触发生成/更新，返回任务状态，不再返回 Bundle URL。
*   `GET /api/model/{dbno}/files`: 返回该数据库所有可用的 Parquet 文件列表。
*   `GET /files/database_models/{dbno}/{filename}`: 静态文件服务。

## 3. 前端开发方案 (plant3d-web)

### 3.1 技术选型
*   **读取库**: 使用 `apache-arrow` (JS版) 或 `duckdb-wasm`。推荐 `apache-arrow`，因为它轻量且能直接与 WebGPU/WebGL 缓冲区交互。

### 3.2 加载逻辑改造 (`useModelGeneration.ts`)

1.  **初始化**:
    *   确定当前项目的 `dbno`。
    *   调用 `GET /api/model/{dbno}/files` 获取文件列表。
2.  **并行加载**:
    *   并发请求 Parquet 文件。
    *   使用 `apache-arrow` 的 `Table.from` 解析数据。
3.  **实例化渲染**:
    *   遍历 Arrow Table 的列。
    *   按 `geo_hash` 分组。
    *   创建 `InstancedMesh` (Three.js)。
    *   *优化*：Arrow 的数据已经是 TypedArray，可以直接复制到 Geometry 的 Attribute Buffer，极大减少 GC 和解析开销。

### 3.3 缓存策略
*   利用浏览器 HTTP Cache (ETag/Last-Modified) 缓存 Parquet 文件。
*   由于是增量文件 (`part_N.parquet`)，旧文件永远不变，天然适合长效缓存。

## 4. 实施计划

### 第一阶段：后端原型
1. [Rust] 引入 `polars` 依赖。
2. [Rust] 实现 `ParquetWriter`，将 `ExportData` 结构体转存为 parquet。
3. [Rust] 实现基于 Polars 的 `refno` 存在性检查。

### 第二阶段：前端适配
1. [Vue] 安装 `apache-arrow`。
2. [Vue] 实现 `ParquetLoader` 类，替代现有的 `AiosPrepackBundleLoader`。
3. [Vue] 调试渲染流程。

### 第三阶段：增量与优化
1. [Rust] 实现 Compact 机制（后台合并碎片文件）。
2. [Vue] 实现视锥体剔除或基于 Parquet 元数据的按需加载。
