# 批注流程与校审流程补强开发计划

## 1. 目标与完成标准

### 目标

把当前“能跑通”的批注与校审流程补强成“可恢复、可审计、可门禁、不会串单”的稳定闭环。

核心目标：

- 批注处理动作不再只停留在前端本地状态。
- 评论、严重度、处理状态都能按 `form_id / task_id` 正确归属。
- `workflow/verify` 和 `workflow/sync` 使用同一套后端真值做流转判断。
- 驳回后设计侧处理批注时，能提前看到未完成问题，不等外部平台流转时才报错。
- 保持现有 `review_records` 快照兼容，不破坏已经保存的数据。

### 完成标准

满足以下结果才算完成：

1. 设计侧、校核侧、审核侧、批准侧都能打开同一个 `form_id` 并看到一致的批注状态、评论、严重度。
2. 批注动作 `已修改 / 不需解决 / 同意 / 驳回` 点击后立即写入后端，不依赖“确认当前数据”才生效。
3. `确认当前数据` 仍保留为正式留痕动作，但不再承担唯一状态真源。
4. `workflow/verify` 能正确阻止未处理、待确认、被驳回的批注继续流转。
5. 两个不同 `form_id` 即使有相同 `annotation_id`，评论也不会串。
6. 非作者不能删除别人的评论，管理员除外。
7. 普通批注面板、审核面板、设计侧批注处理面板修改严重度后结果一致。
8. 前端类型检查、关键单测、后端编译检查、HTTP 合同验证全部通过。

## 2. 关键设计决定

### 2.1 批注处理状态：新增独立后端真源

推荐并采用此方案。

| 方案 | 优点 | 缺点 | 结论 |
|---|---|---|---|
| 继续只写 `review_records` 快照 | 改动小 | 用户没点“确认当前数据”就会丢状态；审计弱 | 不采用 |
| 新增独立状态表，`review_records` 继续做留痕 | 状态可靠；门禁清晰；兼容旧数据 | 需要新增 API 和同步逻辑 | 采用 |
| 全量重构批注系统 | 长期更干净 | 改动过大，风险高 | 不采用 |

新增后端真源表建议名：

`review_annotation_states`

用途：

- 保存每条批注当前处理状态。
- 保存处理动作历史。
- 给 `workflow/verify`、`workflow/sync`、前端恢复统一使用。

---

### 2.2 评论：扩展现有表，不另建新表

推荐并采用此方案。

当前 `review_comments` 已经能保存评论，但缺少上下文。  
本次只扩字段，不重建评论体系。

新增字段：

- `form_id`
- `task_id`
- `workflow_node`
- `review_round`
- `updated_at`
- `deleted`
- `deleted_at`
- `deleted_by`

查询评论时优先使用：

`form_id + task_id + annotation_type + annotation_id + review_round`

旧数据没有这些字段时，保留兼容读取，但新写入必须带上下文。

---

### 2.3 OBB 批注：本轮不纳入正式流转门禁

推荐并采用此方案。

原因：

- 当前代码已经明确把 OBB 排除在提交阻塞之外。
- 已有测试覆盖“只有 OBB 改动时不阻塞提交流转”。
- OBB 更像空间框选/辅助证据，不适合作为本轮强门禁对象。

本轮处理方式：

- OBB 仍可保存、恢复、展示。
- OBB 可进入确认记录。
- OBB 不作为 `workflow/verify` 的阻断条件。
- UI 文案中明确 OBB 是辅助证据，避免用户把它误解成必须处理的正式问题批注。

如果以后业务确认 OBB 是正式问题批注，再单独把它加入门禁类型。

---

### 2.4 评论和批注动作失败时不再静默本地降级

推荐并采用此方案。

当前部分评论、严重度、处理状态操作会在后端失败时保留本地状态。  
这会让用户误以为已经保存，实际后端没有。

本轮规则：

- 校审流程内的评论、严重度、批注处理动作：后端失败则提示失败，不保留虚假的本地成功。
- 非校审普通批注编辑可以继续保留现有本地体验。
- “正式校审留痕”必须以后端成功为准。

### 3.2 新增批注处理状态 API

新增接口：

```http
POST /api/review/annotation-states/apply
Authorization: Bearer <user_token>
Content-Type: application/json
```

请求体：

```json
{
  "formId": "FORM-xxx",
  "taskId": "task-xxx",
  "annotationId": "annotation-xxx",
  "annotationType": "text",
  "action": "fixed",
  "note": "已按意见调整"
}
```

`action` 支持：

```text
fixed
wont_fix
agree
reject
```

响应体：

```json
{
  "success": true,
  "state": {
    "formId": "FORM-xxx",
    "taskId": "task-xxx",
    "annotationId": "annotation-xxx",
    "annotationType": "text",
    "workflowNode": "sj",
    "reviewRound": 2,
    "resolutionStatus": "fixed",
    "decisionStatus": "pending",
    "note": "已按意见调整",
    "updatedById": "SJ",
    "updatedByName": "设计人员",
    "updatedByRole": "designer",
    "updatedAt": 1710000000000,
    "history": []
  }
}
```

新增查询接口：

```http
GET /api/review/annotation-states?form_id=FORM-xxx&task_id=task-xxx
Authorization: Bearer <user_token>
```

用途：

- 前端打开任务时补齐当前批注处理状态。
- 内部任务页面不依赖 `workflow/sync` 时也能恢复状态。

权限规则：

- `fixed / wont_fix`：设计人员、管理员可执行。
- `agree / reject`：校核、审核、批准、管理员可执行。
- 当前用户必须是当前节点负责人，或管理员。
- `task_id` 和 `form_id` 必须匹配。
- 任务已终止、已批准、已删除时禁止写入。

### 3.3 批注门禁改为读取独立状态表

涉及位置：

- `/Volumes/DPC/work/plant-code/plant-model-gen/src/web_api/platform_api/annotation_check.rs`

现有逻辑保留，但数据来源调整为：

1. 读取 `review_records` 中的批注快照，确定有哪些批注。
2. 读取 `review_annotation_states`，覆盖快照里的 `reviewState`。
3. 如果独立状态不存在，再回退到快照内嵌的 `reviewState`。
4. 如果两边都没有状态，按 `open` 处理。
5. 门禁类型保持 `text / cloud / rect`，不包含 OBB。

门禁判断保持现有语义：

| 当前节点 | 状态 | 结果 |
|---|---|---|
| `sj` | 有 `open` 或 `rejected` | 阻止提交，要求处理批注 |
| `sj` | 全部 `fixed / wont_fix / agreed` | 允许送到 `jd` |
| `jd / sh / pz` | 有 `open` 或 `rejected` | 建议退回 |
| `jd / sh / pz` | 有 `pending` | 阻止继续，要求逐条确认 |
| `jd / sh / pz` | 全部 `agreed` | 允许继续流转 |

## 5. HTTP 合同与数据流

### 5.1 批注处理状态合同

```text
前端点击处理动作
  -> POST /api/review/annotation-states/apply
  -> 后端校验 task/form/user/current node
  -> upsert review_annotation_states
  -> 返回 state
  -> 前端更新 toolStore
```

### 5.2 评论合同

```text
前端发送评论
  -> POST /api/review/comments
  -> 后端用 token 写 author
  -> 按 form_id/task_id/annotation_id/review_round 保存
  -> 返回 comment
  -> 前端更新 thread store
```

### 5.3 校审流转门禁

```text
外部平台点击流转
  -> POST /api/review/workflow/verify
  -> 后端读取 review_records
  -> 后端叠加 review_annotation_states
  -> 后端计算批注门禁
  -> passed=true 才允许继续 sync
  -> POST /api/review/workflow/sync
```

### 5.4 快照恢复

```text
前端打开 form_id
  -> POST /api/review/workflow/sync { action: "query" }
  -> 后端聚合 records + comments + states + severity + attachments
  -> 前端回放批注、测量、评论、处理状态、严重度
```

## 10. 风险与处理

| 风险 | 处理方式 |
|---|---|
| 老数据没有独立状态表 | 从 `review_records` 内嵌 `reviewState` fallback |
| 老评论没有 `form_id/task_id` | 查询时兼容旧数据，但新写入必须带上下文 |
| 前端状态和后端状态短暂不一致 | 以后端响应为准，本地只做成功后的同步 |
| 状态 API 成功但确认记录未保存 | 状态已保存；确认记录仍作为正式快照留痕 |
| 严重度历史数据分散在 records 和 severity 表 | `workflow/sync` 聚合时以后端 severity 表优先，records 内嵌为 fallback |
| OBB 用户误解为正式批注 | UI 文案明确“辅助证据”，不进入门禁统计 |
