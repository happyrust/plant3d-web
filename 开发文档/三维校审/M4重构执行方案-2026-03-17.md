# M4 审核工作台重构执行方案

> 生成时间：2026-03-17  
> 目标：将 ReviewPanel 从 1155 行堆叠式组件重构为清晰分区的工作台

## 一、当前问题分析

### 1.1 代码规模
- `ReviewPanel.vue`: 1154 行
- `AnnotationPanel.vue`: 1127 行
- **总计 2281 行**，两个超大组件

### 1.2 ReviewPanel 职责混杂

从代码分析发现以下职责混在一起：

**任务上下文区**（约 100 行）
- 当前任务信息展示
- 标题、模型名、发起人、校核人、审核人
- 当前节点、状态、优先级

**流转动作区**（约 150 行）
- 提交/打回按钮
- `WorkflowSubmitDialog` / `WorkflowReturnDialog`
- 节点权限判断
- 流转加载状态

**确认记录区**（约 200 行）
- 待确认数据统计
- 已确认记录列表
- 确认历史查看
- 删除/清空操作

**辅助数据区**（约 250 行）
- 碰撞查询
- 外部辅助数据查询
- `ReviewAuxData` 组件集成
- 定位/高亮功能

**同步区**（约 100 行）
- 导出/导入功能
- `ReviewDataSync` 组件
- 覆盖策略

**工作流历史区**（约 150 行）
- 历史记录加载
- 时间线展示
- 历史详情

**附件区**（约 100 行）
- 附件列表
- 下载功能

**其他**（约 100 行）
- embed 落点状态
- 模型过滤
- 测量菜单
- 各种工具函数

### 1.3 关键问题

1. **职责不清**：一个组件承担 8+ 个职责
2. **难以维护**：1155 行代码难以理解和修改
3. **测试困难**：无法单独测试各个功能区
4. **复用性差**：功能耦合，无法独立使用
5. **性能问题**：所有功能在一个组件中，影响渲染性能

## 二、重构目标

### 2.1 分区结构

```
ReviewPanel (主容器，约 200 行)
├── TaskContextSection (任务上下文区，约 150 行)
├── WorkflowActionSection (流转动作区，约 150 行)
├── ConfirmedRecordsSection (确认记录区，约 200 行)
├── AuxiliaryDataSection (辅助数据区，约 250 行)
└── DataSyncSection (同步区，约 100 行)
```

### 2.2 设计原则

1. **单一职责**：每个子组件只负责一个功能区
2. **清晰边界**：组件间通过 props/events 通信
3. **可测试性**：每个子组件可独立测试
4. **可复用性**：子组件可在其他场景复用
5. **性能优化**：按需加载，减少不必要的渲染

## 五、风险与应对

### 5.1 风险

1. **重构工作量大**：1155 行代码需要拆分
2. **功能回归风险**：可能引入新 bug
3. **测试覆盖不足**：现有测试可能不完整
4. **时间压力**：预计 7 天，可能延期

### 5.2 应对措施

1. **分阶段进行**：每天完成一个子组件，降低风险
2. **保留备份**：备份原文件，出问题可回滚
3. **增量测试**：每完成一个子组件就测试
4. **并行开发**：可以多人协作，加快进度
