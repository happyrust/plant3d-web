# 三维校审 M3 发起人任务追踪任务单

> 生成时间：2026-03-13  
> 适用阶段：`M3：发起人任务追踪`  
> 对应范围：`DesignerTaskList`、`ResubmissionTaskList`、`TaskReviewDetail`  
> 主设计源：`ui/三维校审/review-designer.pen`  
> 跨角色总览基线：`ui/三维校审/review-flow.pen`（仅用于核对发起人与审核侧流转衔接）  
> 配套文档：`开发文档/三维校审/三维校审重构执行方案.md`、`开发文档/三维校审/三维校审M0页面接口字段映射表.md`

## 1. M3 目标

M3 的目标是建立发起人侧的正式追踪闭环，而不是继续堆叠发起页逻辑。M3 必须完成以下 5 件事：

1. 发起人可以稳定查看自己的任务状态看板。
2. 退回待修改任务有单独大盘，能看见原因与历史。
3. 统一详情页成为发起人查看任务的唯一详情入口。
4. 发起人“修改后再次提交”的动作链清晰可执行。
5. `status/currentNode/returnReason/workflowHistory` 的展示口径统一。

## 3. 前端任务单

### FE-T3-01 发起人状态看板重构

- **目标**：将 `DesignerTaskList` 从 CRUD 列表升级为发起人状态看板。
- **涉及文件**：`src/components/review/DesignerTaskList.vue`、`src/components/review/reviewTaskFilters.ts`、`src/composables/useUserStore.ts`
- **依赖**：`myInitiatedTasks` 数据源；状态分桶规则确认。
- **完成标准**：
  - 稳定展示待审核、已通过、退回待修改三类主状态。
  - 卡片上可直接看到当前节点、责任人、更新时间、退回摘要。
  - 详情统一跳转到 `TaskReviewDetail`。
- **风险/备注**：当前“退回任务”与“待审核”仍依赖前端推断，必须先冻结规则。

### FE-T3-02 退回任务大盘重构

- **目标**：建立专门的 `ResubmissionTaskList`，突出退回原因、退回时间、提交次数与继续修改入口。
- **涉及文件**：`src/components/review/ResubmissionTaskList.vue`、`src/components/review/reviewTaskFilters.ts`
- **依赖**：`returnReason`、`workflowHistory`、提交次数统计规则。
- **完成标准**：
  - 按最近退回时间排序。
  - 每条任务展示最新退回原因、退回时间、累计提交次数。
  - “继续修改”有明确去向，不停留在只读查看。
- **风险/备注**：当前只是过滤展示，没有真正回到修改闭环。

### FE-T3-03 统一详情页重构

- **目标**：将 `TaskReviewDetail` 升级为发起人统一详情页，承载基础信息、时间线、退回记录与再次提交入口。
- **涉及文件**：`src/components/review/TaskReviewDetail.vue`、`src/components/review/reviewPanelActions.ts`
- **依赖**：`reviewTaskGetWorkflow()`；附件、构件、退回历史字段。
- **完成标准**：
  - 详情页最少包含：基础信息、流转时间线、退回记录、附件/构件摘要。
  - 能区分最新退回原因与历史退回记录。
  - 再次提交入口条件明确。
- **风险/备注**：当前 comments tab 并未真正装载任务级评论，M3 要明确是占位还是纳入本期。

### FE-T3-04 发起人再次提交流程闭环

- **目标**：打通发起人收到退回后，查看原因、修改信息、再次提交的完整前端流程。
- **涉及文件**：`src/components/review/DesignerTaskList.vue`、`src/components/review/ResubmissionTaskList.vue`、`src/components/review/TaskReviewDetail.vue`、`src/composables/useUserStore.ts`
- **依赖**：`PATCH /tasks/{taskId}`、`POST /submit` 的时序规则。
- **完成标准**：
  - 再次提交入口位置唯一。
  - 明确允许修改的内容范围。
  - 再次提交后，任务从退回大盘移出并进入待审核。
- **风险/备注**：若本期不改发起编辑页，必须明确临时复用路径。

## 4. 后端任务单

### BE-T3-01 发起人状态口径冻结

- **目标**：冻结发起人看板使用的状态映射规则。
- **依赖**：列表接口、任务模型字段。
- **输出**：明确哪些 `status/currentNode` 组合表示“待审核”“退回待修改”“已通过”。
- **待确认**：历史数据缺失 `currentNode` 或 `returnReason` 的兜底规则。

### BE-T3-02 退回历史数据契约补齐

- **目标**：为退回大盘与详情页提供可读的退回历史结构。
- **依赖**：`/return`、`/workflow`。
- **输出**：每次退回都在 workflow 中保留原因、时间、操作者、节点；明确 `returnReason` 与 workflow comment 的关系。
- **待确认**：`returnReason` 是最新缓存还是正式详情字段。

### BE-T3-03 再次提交流程语义确认

- **目标**：明确发起人修改后再次提交是否复用 `PATCH + /submit`。
- **依赖**：任务更新接口、submit 接口。
- **输出**：再次提交流程时序、前置条件、返回字段。
- **待确认**：是否存在“初次提交”和“重提”的差异语义。

### BE-T3-04 详情补充接口能力确认

- **目标**：保证详情页能拿到构件、附件、退回历史、当前节点、责任人等完整数据。
- **依赖**：详情接口、workflow 接口、评论接口。
- **输出**：M3 详情页数据清单；若任务级评论摘要暂不支持，明确占位策略。
- **待确认**：评论是否继续只按 annotation 维度查询。

## 5. 联调任务单

### INT-T3-01 发起人状态看板联调

- **目标**：验证看板状态分桶与后端真实返回一致。
- **输出**：看板统计数、列表筛选结果、任务详情状态三者一致。
- **必须确认**：`currentNode` 缺失或脏数据的兜底方式。

### INT-T3-02 退回与再次提交流程联调

- **目标**：串通“退回 -> 查看原因 -> 修改 -> 再提交”闭环。
- **输出**：至少一轮 `submit -> return -> submit` 联调样例。
- **必须确认**：再次提交是否必须先 `PATCH` 再 `/submit`。

### INT-T3-03 统一详情页联调

- **目标**：验证从不同入口打开同一任务详情，数据内容一致。
- **输出**：基础信息、退回记录、时间线、附件摘要一致性结论。
- **必须确认**：评论是否纳入 M3 通过标准。

## 6. 验证任务单

### QA-T3-01 发起人核心场景验证

- **覆盖**：查看全部任务、筛选状态、查看退回任务、查看详情、再次提交。
- **通过标准**：发起人能看懂任务状态，并完成退回重提闭环。

### QA-T3-02 数据一致性验证

- **覆盖**：`status/currentNode/returnReason/workflowHistory` 在列表与详情中的一致性。
- **通过标准**：同一任务在多个页面展示口径一致。

### QA-T3-03 回归边界验证

- **覆盖**：M3 不破坏审核入口、任务刷新、基础详情查看能力。
- **通过标准**：M2 与 M4 链路不受影响。

## 7. M3 必须拍板的确认项

1. 发起人状态分桶正式口径。
2. `returnReason` 与 workflow `return.comment` 的关系。
3. 发起人再次提交流程是否统一为 `PATCH + /submit`。
4. 统一详情页本期范围是否包含评论摘要。
5. `reviewerId -> checkerId` 兼容是否继续保留给发起人视图。

## 7.1 设计基线与验收口径

- `DesignerTaskList`、`ResubmissionTaskList`、`TaskReviewDetail` 的页面结构、信息层级与关键交互以 `ui/三维校审/review-designer.pen` 为主设计基线。
- 涉及退回重提与节点流转衔接时，可补充参考 `ui/三维校审/review-flow.pen` 核对跨角色流程关系，但不替代发起人主设计稿。
- 本阶段若提到页面验收、视觉基线或交互对齐，均明确按发起人页面是否对齐 `review-designer.pen` 判定。

## 8. 可留到 M4 / M5 的事项

- 审核工作台内部布局与辅助数据整合
- 悬浮批注工具栏、批注列表、确认动作体系
- 任务级批注意见深度协同与评论大盘
