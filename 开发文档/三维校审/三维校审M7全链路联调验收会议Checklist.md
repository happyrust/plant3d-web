# 三维校审 M7 全链路联调验收会议 Checklist

> 用途：开会快速对齐 + 会后直接进入总联调与验收  
> 对应文档：`开发文档/三维校审/三维校审M7全链路联调验收任务单.md`  
> 设计校验口径：发起侧页面主对照 `ui/三维校审/review-designer.pen`，审核/校核/批准侧页面主对照 `ui/三维校审/review-reviewer.pen`，跨角色流转顺序统一对照 `ui/三维校审/review-flow.pen`

## 一、会议目标

- [ ] 对齐 M7 只做“全链路串联与验收”，不新增二次需求
- [ ] 冻结主流程、异常流程、角色权限、状态机与验收矩阵
- [ ] 明确发起、审核、退回重提、评论、确认、embed、通知的总验收口径
- [ ] 会后可以直接进入总联调执行与结果归档

## 二、会前必读

- [ ] 阅读 `开发文档/三维校审/三维校审M7全链路联调验收任务单.md`
- [ ] 阅读 `开发文档/三维校审/三维校审阶段执行总览.md`
- [ ] 明确发起侧验收主看 `review-designer.pen`，审核侧验收主看 `review-reviewer.pen`，总流程校验再看 `review-flow.pen`
- [ ] 各负责人确认 M1-M6 阶段的已完成项与已知阻塞
- [ ] 测试准备角色账号、样本任务、附件、批注、辅助数据样本

## 三、必须拍板项

- [ ] `currentNode` 是否作为唯一节点真源，`status` 仅作展示
- [ ] 新流程是否只认 `/submit`、`/return`，旧接口仅兼容不纳入新验收
- [ ] `sj / jd / sh / pz` 节点权限边界是否最终冻结
- [ ] 未处理的批注 / 评论 / 确认记录是否阻止提交
- [ ] 退回重提规则是否最终冻结
- [ ] embed 与辅助数据正式参数是否定版

## 四、前端 Owner 清单

- [ ] FE1：梳理发起、审核、退回重提三条入口和跳转/刷新路径
- [ ] FE2：串起 workflow history、评论、确认记录三条数据线的总体验收
- [ ] FE3：验证列表、详情、工作台、embed 落点一致，不混节点、不混状态
- [ ] FE4：补齐关键通知与待办刷新，确认不同角色进入正确页面
- [ ] FE5：整理最终验收矩阵所需页面、账号、操作路径和截图点

## 五、后端 Owner 清单

- [ ] BE1：冻结状态机与接口行为，明确每节点允许动作及动作后状态变化
- [ ] BE2：确保评论、确认记录、附件、辅助数据、通知在同任务上下文闭环
- [ ] BE3：统一关键接口与 WS payload 的最小上下文：`taskId/formId/currentNode/status`
- [ ] BE4：明确旧接口兼容边界，避免新旧流转口径混用
- [ ] BE5：支撑异常链路校验：无权提交、保存失败、参数缺失、WS 断开恢复

## 六、联调 / 测试清单

- [ ] INT1：主流程矩阵：发起 -> 校核 -> 审核 -> 退回/通过 -> 再提交
- [ ] INT2：异常流程矩阵：无权限、评论失败、确认失败、附件成功但任务失败、embed 参数异常、WS 断连
- [ ] QA1：角色权限验收：设计 / 校核 / 审核 / 批准四类角色；发起侧页面对照 `review-designer.pen`，审核侧页面对照 `review-reviewer.pen`
- [ ] QA2：数据一致性验收：列表、详情、工作台、embed、通知中的上下文一致；跨角色流转顺序再对照 `review-flow.pen`
- [ ] QA3：形成最终可执行验收矩阵，并归档结果模板

## 七、风险提醒

- [ ] 风险：`currentNode` 和 `status` 双真源并存，验收结论会失真
- [ ] 风险：workflow / 评论 / 确认记录三条数据线若仍混写，任务回放不可解释
- [ ] 风险：退回重提规则未冻结，主流程无法形成稳定验收口径
- [ ] 风险：embed / 通知 / 附件未纳入同一矩阵，M7 通过后仍可能断链
