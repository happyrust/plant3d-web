# Plant3D-Web 校审流程分析

## 一、系统架构概览

```
┌─────────────────────────────────────────────────────────────────────┐
│                           校审流程架构                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  设计人员 ────► InitiateReviewPanel ────► ReviewTask (提资单)       │
│     │                                          │                    │
│     │                                          ▼                    │
│     │                                    ReviewerTaskList           │
│     │                                     (审核任务列表)            │
│     │                                          │                    │
│     │                                          ▼                    │
│     │                                    ReviewPanel                │
│     │                                     (校审面板)                │
│     │                                          │                    │
│     │                                   ┌──────┴──────┐             │
│     │                                   ▼             ▼             │
│     ◄───────────── approved ◄──── [通过]     [驳回] ────► rejected  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 二、用户角色体系

| 角色 | 枚举值 | 说明 | 权限 |
|------|--------|------|------|
| **系统管理员** | `ADMIN` | 最高权限 | 审核权限 ✅ |
| **项目管理员** | `MANAGER` | 项目级管理 | 审核权限 ✅ |
| **审核人员** | `REVIEWER` | 专职审核 | 审核权限 ✅ |
| **校对人员** | `PROOFREADER` | 校对检查 | 审核权限 ✅ |
| **设计人员** | `DESIGNER` | 模型设计 | 发起提资 ✅ |
| **查看者** | `VIEWER` | 只读查看 | 无操作权限 |

## 三、核心数据结构

### 3.1 提资单 (ReviewTask)

```typescript
type ReviewTask = {
  id: string;
  title: string;
  description: string;
  modelName: string;
  status: 'draft' | 'submitted' | 'in_review' | 'approved' | 'rejected' | 'cancelled';
  priority: 'low' | 'medium' | 'high' | 'urgent';
  requesterId: string;      // 发起人
  reviewerId: string;       // 审核人
  components: ReviewComponent[];  // 包含的构件
  attachments?: ReviewAttachment[];  // 附件
  reviewComment?: string;   // 审核意见
  dueDate?: number;         // 截止日期
  createdAt: number;
  updatedAt: number;
};
```

### 3.2 审核确认记录 (ConfirmedRecord)

```typescript
type ConfirmedRecord = {
  id: string;
  type: 'batch';
  annotations: AnnotationRecord[];       // 文字批注
  cloudAnnotations: CloudAnnotationRecord[];  // 云线批注
  rectAnnotations: RectAnnotationRecord[];    // 矩形批注
  obbAnnotations: ObbAnnotationRecord[];      // OBB批注
  measurements: MeasurementRecord[];     // 测量数据
  confirmedAt: number;
  note: string;                          // 备注
};
```

## 四、状态流转

```
draft ──► submitted ──► in_review ──┬──► approved
                                    │
                                    └──► rejected
                                    │
                                    └──► cancelled
```

| 状态 | 含义 | 显示颜色 |
|------|------|----------|
| `draft` | 草稿 | 灰色 |
| `submitted` | 待审核 | 黄色 |
| `in_review` | 审核中 | 蓝色 |
| `approved` | 已通过 | 绿色 |
| `rejected` | 未通过 | 红色 |
| `cancelled` | 已取消 | 灰色 |

## 六、当前限制

1. **本地存储**：数据仅保存在 localStorage，无后端持久化
2. **模拟用户**：使用 mockUsers 模拟用户系统
3. **无通知机制**：任务状态变化无实时通知
4. **无历史记录**：缺少审核历史追溯功能
5. **无权限校验**：前端角色判断，无后端权限验证

## 七、优化计划

### 7.1 数据库设计 (SurrealDB)

需要创建以下表：
- `review_tasks` - 提资单/审核任务
- `review_records` - 审核确认记录
- `review_history` - 审核历史日志
- `users` - 用户表（如果不存在）

### 7.2 后端 API 设计

```
POST   /api/review/tasks              - 创建提资单
GET    /api/review/tasks              - 获取任务列表
GET    /api/review/tasks/:id          - 获取任务详情
PATCH  /api/review/tasks/:id          - 更新任务
DELETE /api/review/tasks/:id          - 删除任务
POST   /api/review/tasks/:id/approve  - 通过审核
POST   /api/review/tasks/:id/reject   - 驳回审核
GET    /api/review/tasks/:id/history  - 获取审核历史
POST   /api/review/records            - 保存确认记录
GET    /api/review/records/:taskId    - 获取确认记录
```

### 7.3 前端改造

1. 添加 API 调用层
2. 修改 useUserStore 和 useReviewStore 使用后端
3. 添加 WebSocket 实时通知
4. 添加审核历史查看功能
