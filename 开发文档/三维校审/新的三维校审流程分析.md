# 新的三维校审流程分析

> **版本**: 2026-03-24 | **涉及仓库**: `plant3d-web`、`plant-model-gen`
> **配套文档**: [新的三维校审流程简版](./新的三维校审流程简版.md)（适合讲清主线和边界）

## 1. 结论先行

当前“新的三维校审流程”已经不是早期 `plant3d-web` 文档中描述的纯前端 mock/localStorage 方案，而是一个由 `plant3d-web` 与 `plant-model-gen` 协同完成的后端驱动流程：

- `plant3d-web` 负责嵌入入口、工作台 UI、任务选择、三维批注/测量、确认记录、辅助校审数据查询与前端会话维护。
- `plant-model-gen` 负责 JWT 鉴权、任务与工作流接口、确认记录持久化、`form_id` 维度的数据聚合，以及对外/对内的校审同步接口。
- 真正贯穿整条链路的稳定业务主键是 `form_id`，它用于嵌入、辅助校审数据查询、外部工作流同步与跨系统对账。
- 前端工作台内部的任务主键是 `task_id`，它用于前端调用 `/api/review/tasks/*`、打开审核工作台、写入 `review_workflow_history` 等平台内流程。

最容易混淆的点是：当前系统里存在两条不同的“同步链”。

- 一条是**对外暴露的 `form_id` 型同步链**：`/api/review/embed-url`、`/api/review/workflow/sync`、`/api/review/delete`、`/api/review/aux-data`。
- 另一条是**平台内部任务流转后触发的 `task_id` 型通知链**：`/api/review/tasks/{id}/submit|return` 更新任务后，再异步调用 `external_review.workflow_sync_path`。

如果不先分清这两条链路，就会把“平台任务流转”和“跨系统数据同步”误看成同一个接口。

## 2. 两个仓库的职责分工

### `plant3d-web`

核心职责：

- 通过 `reviewGetEmbedUrl()` 申请嵌入地址。
- 在 `DockLayout` 中解析 URL 参数中的 `user_token`、`form_id`、`user_id`，建立前端校审会话。
- 提供 `InitiateReviewPanel`、`ReviewerTaskList`、`ReviewPanel` 等工作台。
- 通过 `/api/review/tasks*` 与 `/api/review/records*` 驱动任务创建、流转与确认记录持久化。
- 通过 `/api/review/aux-data` 拉取辅助校审数据。
- 通过 WebSocket 监听记录保存、任务更新、评论新增等实时事件。

关键文件：

- `plant3d-web/src/components/DockLayout.vue`
- `plant3d-web/src/components/review/InitiateReviewPanel.vue`
- `plant3d-web/src/components/review/ReviewerTaskList.vue`
- `plant3d-web/src/components/review/reviewerTaskListActions.ts`
- `plant3d-web/src/components/review/ReviewPanel.vue`
- `plant3d-web/src/components/review/ReviewAuxData.vue`
- `plant3d-web/src/composables/useUserStore.ts`
- `plant3d-web/src/composables/useReviewStore.ts`
- `plant3d-web/src/api/reviewApi.ts`

### `plant-model-gen`

核心职责：

- 提供 `/api/auth/token`、`/api/auth/verify` 管理 JWT。
- 提供 `/api/review/tasks*`、`/api/review/records*` 作为平台内部任务/记录 API。
- 提供 `/api/review/embed-url`、`/api/review/workflow/sync`、`/api/review/delete`、`/api/review/cache/preload` 作为对外集成接口。
- 提供 `/api/review/aux-data` 作为辅助校审数据代理/整合接口。
- 在任务流转后异步通知外部系统。

关键文件：

- `plant-model-gen/src/web_api/review_api.rs`
- `plant-model-gen/src/web_api/model_center_client.rs`
- `plant-model-gen/src/web_api/review_integration.rs`
- `plant-model-gen/src/web_api/jwt_auth.rs`
- `plant-model-gen/shells/test_workflow_e2e.sh`

## 3. 核心业务对象

### `form_id`

这是当前新流程里最重要的跨系统业务锚点。

用途包括：

- 嵌入地址生成与回放
- 当前任务 lineage 识别
- 辅助校审数据查询
- 外部工作流同步
- 模型清单、审批意见、附件的聚合查询

### `task_id`

这是平台内部任务主键，主要服务于前端工作台与内部任务接口：

- `/api/review/tasks/{id}`
- `/api/review/tasks/{id}/submit`
- `/api/review/tasks/{id}/return`
- `/api/review/tasks/{id}/workflow`
- `/api/review/records/by-task/{task_id}`

### `current_node`

当前已对齐为四节点：

- `sj`：编制/设计
- `jd`：校对
- `sh`：审核
- `pz`：批准

### `status`

当前任务状态以平台任务为准，主要出现：

- `draft`
- `submitted`
- `in_review`
- `approved`

### `ConfirmedRecord`

由审核工作台中的批注/测量“确认数据”动作产生，存储在 `review_records`，是三维审核过程中的阶段性确认快照。

## 4. 当前真实流程

### 4.1 嵌入入口与认证建立

#### 4.1.1 合同上的理想形态

1. 前端或外部系统通过 `/api/review/embed-url` 获取嵌入地址。
2. 后端会复用或生成 `form_id`，同时生成 JWT。
3. 返回的数据中除了 URL，还包含：
   - `query.form_id`
   - `token`
   - `lineage.form_id`
   - `lineage.task_id`
   - `lineage.current_node`
   - `lineage.status`
   - `task`（若该 `form_id` 已绑定既有任务）
4. `plant3d-web` 的 `DockLayout` 在嵌入模式下读取 URL 里的 `user_token` 和 `form_id`。
5. `DockLayout` 会把 `user_token` 存入 `review_auth_token`，再调用 `/api/auth/verify` 验证，会话成立后再初始化用户与落点面板。

这一步的结果是：前端后续所有 `/api/review/tasks*`、`/api/review/records*` 请求都可以自动携带 `Authorization: Bearer <token>`。

#### 4.1.2 2026-03-24 在 PMS 页面里抓到的实际运行逻辑

按照 `编校审交互接口设计.md` 的约束，PMS 理应统一使用**同一次** `embed-url` 返回的 `form_id`、`token` 与 `url`。但这轮通过 Chrome DevTools 从 PMS 页面抓到的运行态表明，当前实现已经偏成了“两段获取、前端拼装”：

1. `FormLoad(add)` 时，当前 PMS 单据里的 `ModelUrl`、`ModelFormId` 还是空。
2. PMS 页面在新增态调用 `POST /HD/GetZyModelUrl`。
3. 调用成功后，页面执行刷新并跳转到 `EditForm`。
4. `FormLoad(edit)` 返回后，当前单据记录中会出现：
   - `ModelUrl=/review/3d-view?...&form_id=FORM-...`
   - `ModelFormId=FORM-...`
5. 页面随后又单独调用 `POST /HD/GetZyModeInfo` 获取：
   - `ip_addr`
   - `token`
6. 页面前端最终把两段数据拼成 iframe `src`：

```text
root_url + currow.ModelUrl + "&user_token=" + token + "&user_id=" + sessiondata.HumanCode
```

这个实际运行逻辑有两个关键特征：

- `query.form_id` 来自 PMS 当前记录里的 `ModelUrl/ModelFormId`
- `user_token` 来自 `GetZyModeInfo`

而当前抓到的 `GetZyModeInfo` 请求只传：

```json
{
  "username": "SJ",
  "project": "AvevaMarineSample",
  "role": "SJ"
}
```

并**没有显式传 `form_id`**。这意味着当前 PMS 页面层并不是“直接消费一次 `embed-url` 的原子响应”，而是把：

- **单据层 form_id**
- **会话层 token**

拆成两条链路，再在浏览器里自己拼接。

这也是为什么 2026-03-24 的多轮联调里会反复看到：

- PMS 当前记录中的 `ModelFormId = A`
- iframe `query.form_id = B`
- `token.claims.form_id = C`

三者不一致的情况。  
因此，当前 PMS 运行态已经不能简单表述为“平台统一使用 `embed-url` 返回的 `form_id`”，而应理解为：

> **PMS 后端大概率做了 `embed-url` 等价能力来生成并保存单据级 `ModelFormId`，但页面打开时又额外请求 token，并在前端重新拼装嵌入地址。**

### 4.2 设计侧发起编校审

入口：

- `panel.initiateReview` 命令
- `DockLayout` 打开 `initiateReview` 面板

真实流程位于 `InitiateReviewPanel.vue`：

1. 读取嵌入上下文与 `externalWorkflowMode`。
2. 用户选择构件、填写数据包名称、描述、附件。
3. 调用 `userStore.createReviewTask()`，前端最终走到 `/api/review/tasks`。
4. 后端创建 `review_tasks` 记录，初始化：
   - `form_id`
   - `status='draft'`
   - `current_node='sj'`
   - `workflow_history=[]`
5. 后端同时把构件 `refNo` 写入 `review_form_model`，建立 `form_id -> model_refno` 关系。

这里有一个非常重要的分支：

- **external 模式**：只创建编校审数据，不自动推进内部节点。
- **manual/internal 模式**：创建成功后立即调用 `submitTaskToNextNode(task.id, '发起编校审')`。

这意味着当前系统默认更偏向“外部流程驱动”，而不是“前端独立闭环驱动”。

### 4.3 审核侧进入工作台

入口：

- `review.start` 命令
- `DockLayout` 打开 `reviewerTasks`

在 `reviewerTaskListActions.ts` 中，审核侧链路是：

1. 如果任务还是 `submitted`，先调用 `/api/review/tasks/{id}/start-review`。
2. 刷新任务列表。
3. 基于 `task.id` 或 `formId` 重新定位任务，避免打开旧快照。
4. `reviewStore.setCurrentTask(taskToOpen)`。
5. 先显示 `reviewerTasks`，再调度打开 `panel.review`。

这一步表明，审核工作台打开时不是直接靠路由参数，而是靠当前任务上下文对象驱动。

### 4.4 审核工作台

`ReviewPanel.vue` 是当前审核流程的核心页面，主要做四件事：

1. 根据 `currentTask` 过滤三维模型，只显示当前任务相关构件。
2. 加载当前任务的工作流历史与确认记录。
3. 提供批注、云线、矩形、测量等审核工具。
4. 执行“确认数据”“提交到下一节点”“驳回到指定节点”。

其中“确认数据”流程是：

1. 汇总临时批注、云线、矩形、测量。
2. 调用 `reviewStore.addConfirmedRecord()`。
3. 前端走 `/api/review/records`。
4. 后端写入 `review_records`。
5. 前端清空临时工具状态，只保留确认记录。

这说明当前新流程已经将“审核过程中的操作痕迹”分成两层：

- 临时编辑态：工具面板中的即时批注/测量
- 已确认态：`ConfirmedRecord`

### 4.5 辅助校审数据

`ReviewAuxData.vue` 已把辅助校审数据接进当前任务上下文：

- 使用 `project_id`
- 使用当前任务 `formId`

- 使用当前任务构件 `refNo`
- 使用 `requester_id`

请求路径：

- 前端：`/api/review/aux-data`
- 后端：`review_integration.rs` 中的 `get_aux_data`

当前实现特征：

- 依赖 `UCode` / `UKey` 请求头
- 优先从 `collision_events` 查真实碰撞
- 查不到则回退到 mock 碰撞数据
- `quality` / `otverification` / `rules` 目前还是占位结构

所以“辅助校审数据链”已经接通，但只有碰撞数据相对更接近可用状态。

### 4.6 提交与驳回

前端在 `ReviewPanel.vue` 中最终调用：

- `userStore.submitTaskToNextNode()`
- `userStore.returnTaskToNode()`

而 `useUserStore.ts` 在 `USE_BACKEND=true` 时实际走的是：

- `/api/review/tasks/{id}/submit`
- `/api/review/tasks/{id}/return`

后端 `review_api.rs` 的真实行为：

#### 提交到下一节点

1. 读取当前任务。
2. 按 `current_node` 做权限校验。
3. 计算下一个节点与目标状态：
   - `sj -> jd` 时状态为 `submitted`
   - `jd -> sh`、`sh -> pz` 时状态为 `in_review`
   - `pz` 再提交则视为最终批准，状态变 `approved`
4. 更新 `review_tasks.current_node/status`。
5. 写入 `review_workflow_history`。
6. 异步触发 `notify_workflow_sync_async(task_id, action, operator_id, comment)`。

#### 驳回到指定节点

1. 读取当前任务。
2. 校验当前节点是否有驳回权限。
3. 校验目标节点是否合法。
4. 更新：
   - `current_node`
   - `status`
   - `return_reason`
5. 写入 `review_workflow_history`。
6. 异步触发外部通知。

这条链路说明：**平台内任务状态推进的主入口是 `/api/review/tasks/{id}/submit|return`，不是 `/api/review/workflow/sync`。**

### 4.7 实时刷新

`useReviewStore.ts` 在 `setCurrentTask(task)` 后会：

1. 加载确认记录
2. 加载审核历史
3. 根据 `task.reviewerId` 建立 WebSocket 连接

当前监听的消息类型包括：

- `record_saved`：刷新确认记录
- `task_updated` / `task_approved` / `task_rejected`：刷新历史
- `comment_added`：转发给注册回调
- `pong`：心跳

所以当前新流程已经具备“多人协同下的任务状态与记录刷新”基础能力。

## 5. 两条同步链的区别

### 5.1 `form_id` 型外部集成链

这条链主要在 `model_center_client.rs` 与 `review_integration.rs`。

入口包括：

- `/api/review/embed-url`
- `/api/review/workflow/sync`
- `/api/review/delete`
- `/api/review/cache/preload`
- `/api/review/aux-data`

特点：

- 面向外部系统或嵌入侧
- 主键以 `form_id` 为核心
- 目标是把“这个业务单据”在平台和外部系统之间对齐
- `workflow/sync` 的 `action=query` 只读查询
- `workflow/sync` 的非查询动作会写 `review_opinion`

### 5.2 `task_id` 型平台内部流转链

这条链主要在 `review_api.rs`。

入口包括：

- `/api/review/tasks`
- `/api/review/tasks/{id}/submit`
- `/api/review/tasks/{id}/return`
- `/api/review/tasks/{id}/workflow`
- `/api/review/records/*`

特点：

- 面向 `plant3d-web` 审核工作台
- 主键以 `task_id` 为核心
- 负责平台任务状态推进、工作流历史、确认记录管理
- 提交/驳回后会异步调用外部通知

### 5.3 当前系统中的关系

两条链的关系不是二选一，而是“内外分层”：

- **平台内部真实操作**：围绕 `task_id`
- **跨系统追踪与对账**：围绕 `form_id`

而 `form_id` 和 `task_id` 的桥梁，就是 `review_tasks.form_id`。

## 6. 当前后端存储模型

如果只回答“数据最终存到哪里”，当前实际是一个**按 `form_id` 与 `task_id` 分层**的存储模型：

### 6.1 `review_tasks`：平台内部任务主表

创建编校审任务时，后端会在 `review_tasks` 中写入：

- `id = task-...`
- `form_id`
- `title`
- `description`
- `model_name`
- `status = 'draft'`
- `current_node = 'sj'`
- `checker_id / approver_id / reviewer_id`
- `components`
- `attachments`

这张表的主键是 `task_id`，但它同时保存 `form_id`，作为平台内部任务与外部单据的桥接点。

### 6.2 `review_form_model`：`form_id -> model_refno` 关联表

创建任务成功后，后端会把构件列表中的唯一 `refNo` 同步写入 `review_form_model`：

- `form_id`
- `model_refno`
- `created_at`

因此，`workflow/sync?action=query` 能够按 `form_id` 返回 `models`。

### 6.3 `review_attachment`：附件聚合表 + 文件落盘

真正的附件上传接口会：

1. 把文件写入 `assets/review_attachments/`
2. 再写 `review_attachment`

其聚合维度同样是 `form_id`，同时保留 `model_refnos`，用于向外同步当前单据下的附件清单。

### 6.4 `review_opinion`：跨系统审批意见表

当 `/api/review/workflow/sync` 以 `active / agree / return / stop` 等非 `query` 动作写入时，后端会按 `form_id` 创建 `review_opinion`：

- `form_id`
- `model_refnos`
- `node`
- `seq_order`
- `author`
- `opinion`
- `created_at`

所以它本质上是**单据维度**的审批意见表，而不是内部任务维度的表。

### 6.5 `review_records`：审核工作台确认记录表

审核工作台里的“确认数据”动作会写入 `review_records`，其主关联键是 `task_id`，用于保存：

- 文字批注
- 云线
- 矩形
- OBB
- 测量
- 备注

这部分更偏平台内部工作台数据。

### 6.6 `review_workflow_history`：平台内部流转历史

`/api/review/tasks/{id}/submit|return` 更新任务节点时，会写入 `review_workflow_history`：

- `task_id`
- `node`
- `action`
- `operator_id`
- `operator_name`
- `comment`
- `timestamp`

因此“工作台历史”和“对外意见同步”在后端是两套不同表：

- 平台内流转：`review_tasks + review_workflow_history`
- 对外聚合同步：`review_form_model + review_opinion + review_attachment`

### 6.7 2026-03-24 的新建单据追踪结果

这轮通过 PMS 入口重新新建了一张真实单据，数据包名为：

- `TRACE-STORAGE-1774326200`

随后对运行中的 `web_server` 做了真实接口追踪，结果是：

1. `GET /api/review/tasks` 中存在一条新任务：
   - `task_id = task-9dc9b0ab-352c-4e37-87cc-23d8b9dc31af`
   - `form_id = FORM-83FA6B57325A`
   - `status = draft`
   - `current_node = sj`
2. 对 `form_id=FORM-83FA6B57325A` 调 `POST /api/review/workflow/sync action=query`，能查到：
   - `models = ["24381_145018"]`
3. PMS 某条可读记录里也能看到：
   - `ModelFormId = FORM-83FA6B57325A`

这说明：

- **任务主表已经存下来了**
- **构件关联已经按 `form_id` 存下来了**
- **PMS 某一层记录里也保存了同一个 `form_id`**

但同一次页面打开时，当前 iframe 实际出现的：

- `query.form_id`
- `token.claims.form_id`

却又是另外两个值，而且对它们做 `workflow/sync query` 时返回为空。  
所以当前问题不是“完全没存”，而是：

> **保存链已经把任务和构件关联落到正确的 `form_id` 上了，但 PMS 打开页时没有稳定回到这条已保存单据。**

## 7. 代码与文档的一致性判断

### 一致的部分

- 四节点角色编码已经基本统一为 `sj/jd/sh/pz`
- `form_id` 已成为稳定 lineage
- 前端工作台、后端任务 API、辅助数据接口、E2E 脚本已经能串出完整新链路
- `test_workflow_e2e.sh` 与代码实现的大方向一致：先 `embed-url`，再建任务，再推进任务，再做 workflow 数据验证

### 已经过时或容易误导的部分

- `plant3d-web/开发文档/三维校审/校审流程分析.md` 仍然保留了明显的 mock/localStorage 时代描述，不足以代表当前真实实现
- 新流程已经是“后端优先 + JWT + form_id lineage + WebSocket”的组合，不应再用旧文档当作唯一依据
- 对外合同中“PMS 统一使用 `embed-url` 返回的 `form_id`”这句话，在当前运行态下也已经不能直接当成事实；更准确的说法应该是“PMS 当前单据层保存了一个 `form_id`，但页面层打开嵌入页时仍存在拆链与重拼装”

## 8. 当前风险与待确认项

### 8.1 `embed-url` 里存在临时关闭 token 校验的痕迹

`model_center_client.rs` 的 `get_embed_url()` 中明确写了：

- `TODO: 暂时关闭 token 验证，后续需恢复`

这意味着嵌入地址获取接口当前仍有安全补齐空间。

### 8.2 `cache/preload` 还是占位实现

`/api/review/cache/preload` 当前只是返回：

- `code: 0`
- `message: accepted`
- `task_id: cache_<project_id>`

真正的模型中心缓存触发逻辑尚未落地。

### 8.3 `aux-data` 只有碰撞数据相对成型

当前：

- `collision` 可查库，也可 mock
- `quality` / `otverification` / `rules` 还是占位

所以辅助校审数据链路还没有完全达到文档中列出的完整能力。

### 8.4 `externalWorkflowMode` 默认值偏外部驱动

`InitiateReviewPanel.vue` 默认把 `externalWorkflowMode` 设为 `true`，只有显式切到 `manual/internal` 才会进入“创建后自动提交”路径。

如果使用方不理解这个模式差异，就会出现“任务创建成功但节点没有推进”的误判。

### 8.5 “sync” 概念容易被误解

当前至少有两个语义不同的 sync：

- 对外公开的 `/api/review/workflow/sync(form_id...)`
- 内部触发的 `external_review.workflow_sync_path(task_id...)`

建议后续文档单独命名为：

- “单据数据同步”
- “任务状态通知”

避免统称为 `workflow sync`。

### 8.6 PMS 页面层存在 `form_id` 分叉风险

截至 2026-03-24 的运行态证据表明：

- `GetZyModelUrl` 会让 PMS 当前单据保存 `ModelUrl / ModelFormId`
- `GetZyModeInfo` 会单独返回 `token`
- 页面前端会把两者重新拼成 iframe 地址

而 `GetZyModeInfo` 的页面请求入参里没有显式 `form_id`。  
这会导致：

- `PMS 记录里的 ModelFormId`
- `iframe query.form_id`
- `token.claims.form_id`

可能出现分叉，最终让 `plant3d-web` 在 `DockLayout -> /api/auth/verify` 这条链上拿到不一致的 lineage。

### 8.7 多用户从 PMS 继续流转时，当前不一定能回到同一任务

截至 2026-03-24 的 CDP 联调结果：

- 设计侧通过 PMS 入口创建编校审任务可以成功
- 但换用户重新进入 PMS 后，列表未必能稳定回到刚创建的那条任务
- 在这种情况下，下一位用户打开的可能是一个新的设计态嵌入页，而不是当前任务的审核工作区

这会直接影响是否能真实触发：

- `workflow/sync active/agree/return`
- reviewer/校对工作区打开
- 同一 `form_id` 下的意见聚合

## 9. 建议的统一理解

如果只用一句话描述当前新流程，可以表述为：

> 先通过 `form_id` 建立跨系统单据 lineage，再在平台内部围绕 `task_id` 完成三维审核工作台操作，最后把任务状态变化和意见数据按需要同步回外部系统。

换成更落地的步骤就是：

1. 外部或前端获取嵌入地址，拿到 `form_id + JWT`
2. 前端建立嵌入会话并打开正确的落点面板
3. 设计侧创建编校审任务，任务绑定 `form_id`
4. 审核侧从任务列表进入工作台
5. 在工作台中完成批注、测量、确认记录、辅助数据查看
6. 通过 `/api/review/tasks/{id}/submit|return` 推进平台内部节点
7. 后端写历史、发通知，并通过 `form_id` 维度向外提供聚合查询与同步能力
