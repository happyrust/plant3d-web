## 1. 术语表（权威）

| 术语 | 类型 | 生成方 | 稳定性 | 说明 |
| --- | --- | --- | --- | --- |
| `annotationId` | string | 当前后端 / 前端旧逻辑 | **批次内稳定，跨快照可能变** | 存储实体主键，保留兼容 |
| `annotationKey` | string | 前端 v1（§3）+ 后端 v2 | **跨快照/跨恢复稳定** | 评论与批注的业务锚点 |
| `taskId` | string | 后端 | 任务生命周期稳定 | 平台内任务主键 |
| `formId` | string | 外部系统（PMS） | 表单生命周期稳定 | 嵌入入口主键 |
| `workflowNode` | enum: `sj` \| `jd` \| `sh` \| `pz` | 后端 | 节点稳定 | 工作流阶段（设/校/审/批） |
| `reviewRound` | integer, 从 1 开始 | 后端 | 轮次稳定 | 每次**退回后重新提交**自增 1 |
| `annotationType` | enum: `text` \| `cloud` \| `rect` \| `obb` | 前端 | 类型稳定 | 批注图元类型 |
| `source` | enum + open string | 前端 | 稳定 | `ReviewSnapshot.source`：`task_records` / `workflow_sync` / `import_package` / `offline_cache` / `history_replay` |

**命名约束**：
- 禁止用 `currentNode` 指代 `workflowNode`（语义偏差：currentNode 是"当前正在哪个节点"，workflowNode 是"这条记录归属的节点"）。
- 禁止 UI 层再自造"阶段"/"环节"/"步骤" 等别名词。
- WebSocket/REST 所有新字段必须使用本表术语。

## 3. `annotationKey` 生成策略

### v1（前端临时，兼容期）

```
annotationKey = sha1_hex(
  [annotationType, taskId, normalizedGeometrySignature, normalizedContent]
    .map(canonicalize)
    .join('\u001F')   // ASCII unit separator
)[:16]
```

- `canonicalize`：trim + 小写 + NFC 归一化 + 数值保留 4 位小数（`toFixed(4)`）。
- `normalizedGeometrySignature`：
  - `text` → `"x,y,z"`（批注锚点坐标）
  - `cloud` → `points.length + sha1(points.flat().toFixed(4))` 前 8 位
  - `rect` / `obb` → `center + size`
- 结果长度 16 hex = 64 bit 冲突概率，1k 批注下可忽略。

### v2（后端权威，长期）

- 后端写库时生成 UUID v7（时序 + 随机），写回响应。
- 前端收到后覆盖 v1 的值，后续 PATCH/POST 以后端值为准。
- 兼容期：API 响应可同时含 `annotationKey`（后端）与 `annotationKeyV1`（前端计算，仅用于**审计对比**）。

### 迁移

- Phase D 启用前：仅 v1，评论与批注按 v1 归并。
- Phase D 启用后：v2 优先，v1 保留但不参与归并。
- 双写期 ≥ 2 个发版周期。

## 4. `reviewRound` 生命周期

### 协议

- **批注级字段**：每条 `AnnotationEntity` 带 `reviewRound`。
- **评论级字段**：每条 `CommentThread` item 带 `reviewRound`。
- **任务级无 round**：`taskId` 跨所有轮次稳定，不随 round 变化。

### 自增规则

- 初始：校审发起时 `reviewRound = 1`。
- **退回**（`jd/sh/pz` 节点点"打回编制人"）时：后端原子自增任务下的 `currentRound`，下一次创建的批注/评论继承新 round。
- **已有批注不迁移 round**：即 round=1 的批注在 round=2 里仍然是 round=1，过滤器需支持"仅本轮" / "累计全部"。

### 前后端职责

- **后端**：保存 `task.currentRound`，创建批注/评论时自动填充 `reviewRound = task.currentRound`。
- **前端**：不自行计算 round，只读。

## 5. Feature Flag 规范

### 命名规则

```
REVIEW_<PHASE>_<FEATURE>_<STAGE>
```

- `PHASE`：`A`/`B`/`C`/`D`/`E`/`F`/`G`/`H`
- `FEATURE`：小写短语，如 `snapshot_layer`/`comment_thread_store`
- `STAGE`：`SHADOW`（仅跑不影响渲染）/`DUAL_READ`（新旧并行读）/`CUTOVER`（切到新）

### 示例

| Flag | 阶段 | 默认 | 说明 |
| --- | --- | --- | --- |
| `REVIEW_B_SNAPSHOT_LAYER_SHADOW` | B | `false` | 开启后所有 restore 同时产出 `ReviewSnapshot` 但不消费 |
| `REVIEW_B_SNAPSHOT_LAYER_CUTOVER` | B | `false` | 开启后 UI 直接消费 snapshot |
| `REVIEW_C_COMMENT_THREAD_STORE_DUAL_READ` | C | `false` | `ReviewCommentsTimeline` 同时读 annotation.comments 与 thread store 对比 |
| `REVIEW_C_COMMENT_THREAD_STORE_CUTOVER` | C | `false` | 切换为仅读 thread store |
| `REVIEW_D_ANNOTATION_KEY_V2` | D | `false` | 启用后端权威 key |
| `REVIEW_E_DUAL_LAYER_VIEWER` | E | `false` | viewer draft/confirmed 分层 |
| `REVIEW_F_COMMENT_SYNC_V2` | F | `false` | commentSyncService 接管 WebSocket |

### 回滚路径

每个 Flag 必须配套：
1. **关闭即回退**：关 flag 不需要部署回退。
2. **灰度分桶**：按 `taskId hash % 100` 灰度到 X%。
3. **降级开关**：前端检测 `localStorage['review.force_legacy']=1` 强制回滚（用户排障）。

## 6. 性能基线

Phase B 结束前必须有以下基线数据（随大型模型+批注数集压测）：

| 指标 | 目标 | 测量点 |
| --- | --- | --- |
| 首次任务切换 restore 耗时 | < 500ms (100批注) / < 2s (1000批注) | `confirmedRecordsRestore` 进 → scene ready |
| workflow/sync 恢复耗时 | < 1s (P95) | 网络请求成 → 页面可交互 |
| 评论拉取 | < 200ms (50评论/批注) | `reviewCommentGetByAnnotation` 完成 |
| 切换任务内存增长 | < 20MB/任务 | `performance.memory.usedJSHeapSize` |
| WebSocket comment_added 到 UI 刷新 | < 300ms | 消息到达 → DOM 更新 |

未达标需在对应 Phase 结束前加缓存/分页/增量策略。

## 7. 关键测试用例清单（补强）

除原文 §7.3 外，**必须新增**：

### 跨轮次评论归并
- 用例：同一 `annotationKey` 在 round=1 + round=2 各有 3 条评论，UI 切换"仅本轮"/"累计全部"显示正确。
- 位置：`src/review/services/__tests__/commentRoundMerge.test.ts`

### 断网重连收敛
- 用例：A 发评论时网络断 5s 后恢复，B 收到的评论不重复、顺序正确。
- 依赖：WebSocket mock + msg seq + idempotencyKey。

### 三端一致性
- 用例：同一 `taskId` 从平台内 / workflow/sync / 离线导入包三个入口打开，`ReviewSnapshot` 字段一致（annotations.length / comments.length / measurements.length / meta.hash）。

### draft/confirmed 隔离
- 用例：Phase E 启用后，在已确认任务里新增 draft 批注，导出与 confirmed 不混。

### 大规模性能
- 用例：1000 批注 + 3000 评论的任务，restore < 2s，内存增长 < 30MB。

## 8. Phase H 前置项

将 **client-side event log** 前置到 Phase C：

- 每次评论/批注变更在前端产生 event 记录（内存环形 buffer，最近 500 条）。

- 位置：`src/review/services/eventLog.ts`（Phase C 落地）。
