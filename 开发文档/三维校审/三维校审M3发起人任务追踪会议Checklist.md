# 三维校审 M3 发起人任务追踪会议 Checklist

> 用途：开会快速对齐 + 会后直接派单  
> 主设计源：`ui/三维校审/review-designer.pen`  
> 跨角色总览基线：`ui/三维校审/review-flow.pen`（仅用于补充总流程上下文）  
> 对应文档：`开发文档/三维校审/三维校审M3发起人任务追踪任务单.md`

## 一、会议目标

- [ ] 对齐 M3 只做“发起人任务追踪”，不扩到 M4 / M5
- [ ] 拍板状态分桶、退回口径、再次提交流程
- [ ] 明确发起人看板、退回大盘、统一详情页三块边界
- [ ] 会后可以直接按 owner 派单开工

## 二、会前必读

- [ ] 阅读 `开发文档/三维校审/三维校审M3发起人任务追踪任务单.md`
- [ ] 阅读 `开发文档/三维校审/三维校审重构执行方案.md`
- [ ] 阅读 `开发文档/三维校审/三维校审M0页面接口字段映射表.md`
- [ ] 会前过一遍 `ui/三维校审/review-designer.pen`，明确发起人看板、退回大盘、统一详情页的主设计基线
- [ ] 前端确认范围页面：`DesignerTaskList` / `ResubmissionTaskList` / `TaskReviewDetail`
- [ ] 后端准备状态字段、退回历史、再次提交流程接口现状

## 三、必须拍板项

- [ ] 待审核 / 退回待修改 / 已通过的正式分桶规则
- [ ] `returnReason` 与 workflow 里的 `return.comment` 的关系
- [ ] 再次提交是否统一按 `PATCH + /submit` 执行
- [ ] 统一详情页本期是否纳入评论摘要
- [ ] 历史脏数据缺失 `currentNode / returnReason` 的兜底策略

## 四、前端 Owner 清单

- [ ] FE1：重构 `DesignerTaskList` 为发起人状态看板
- [ ] FE2：重构 `ResubmissionTaskList`，突出退回原因、时间、提交次数、继续修改入口
- [ ] FE3：升级 `TaskReviewDetail` 为统一详情入口
- [ ] FE4：串通“查看原因 -> 修改 -> 再次提交”的唯一操作路径
- [ ] FE5：统一列表与详情页的状态展示口径

## 五、后端 Owner 清单

- [ ] BE1：冻结发起人看板状态映射规则
- [ ] BE2：补齐退回历史契约：原因、时间、操作者、节点
- [ ] BE3：明确再次提交接口语义、前置条件和返回字段
- [ ] BE4：确认详情页字段清单是否覆盖构件、附件、退回历史、责任人
- [ ] BE5：明确评论能力若本期不支持时的占位策略

## 六、联调 / 测试清单

- [ ] INT1：验证看板分桶与详情展示一致
- [ ] INT2：跑通至少一轮 `submit -> return -> submit` 闭环样例
- [ ] INT3：验证不同入口进入同一详情页的数据一致性
- [ ] QA1：验证发起人查看全部任务、退回任务、详情、再次提交主场景
- [ ] QA2：验证 `status/currentNode/returnReason/workflowHistory` 一致性

## 七、风险提醒

- [ ] 风险：退回任务与待审核仍靠前端推断，会持续返工
- [ ] 风险：再次提交若接口语义不清，前端入口无法收口
- [ ] 风险：评论摘要范围不拍板，详情页会继续膨胀
- [ ] 风险：历史数据字段缺失会直接影响状态可信度

## 八、设计验收提醒

- [ ] 发起人看板、退回大盘、统一详情页的视觉与交互验收统一对照 `ui/三维校审/review-designer.pen`
- [ ] 若需要讨论跨角色流转衔接，可补充参考 `ui/三维校审/review-flow.pen`，但不替代发起人主设计稿
- [ ] 会议结论中的页面验收口径，明确写成“发起人页面以 `review-designer.pen` 为准”
