# 三维校审 M2 审核入口统一化任务单

> 生成时间：2026-03-13  
> 适用阶段：`M2：审核入口统一化`  
> 对应范围：`ReviewerTaskList`、`WorkflowSubmitDialog`、`WorkflowReturnDialog`  
> 主设计源：`ui/三维校审/review-reviewer.pen`  
> 总览基线：`ui/三维校审/review-flow.pen`（仅用于跨角色流程与节点关系对照）  
> 配套文档：`开发文档/三维校审/三维校审重构执行方案.md`、`开发文档/三维校审/三维校审M0页面接口字段映射表.md`

## 1. M2 目标

M2 的目标是建立稳定、唯一、可联调的审核入口，而不是一次性完成工作台或发起人视角重构。M2 必须完成以下 5 件事：

1. 审核人/校核人的“我的待办”任务池有统一口径。
2. 从待办列表进入处理页只有一条主路径。
3. 提交与打回动作统一通过标准弹窗入口触发。
4. 节点权限、按钮文案、状态筛选规则统一。
5. `/submit`、`/return` 成为 M2 主链路，旧接口仅保留兼容边界。

## 3. 前端任务单

### FE-T2-01 任务池口径收口

- **目标**：统一 `ReviewerTaskList` 的“我的待办”定义，明确只由 `currentNode + 当前登录人角色/ID + status` 决定。
- **涉及文件**：`src/components/review/ReviewerTaskList.vue`、`src/composables/useUserStore.ts`、`src/types/auth.ts`
- **依赖**：M0 中 `checkerId`、`approverId`、`reviewerId` 兼容口径；后端稳定返回 `currentNode`、`status`。
- **完成标准**：
  - 列表数据源只保留一套待办筛选规则。
  - 校核待办只看 `jd`，审核/批准待办只看 `sh/pz`。
  - 组件内不再额外补充一套临时判定规则。
- **风险/备注**：当前 `pendingReviewTasks` 仍混用 `checkerId || reviewerId`，兼容字段未收口前容易出现错分配。

### FE-T2-02 筛选项与状态展示统一

- **目标**：统一任务列表筛选项、状态文案与展示态，避免“status 过滤”和“currentNode 权限”各说各话。
- **涉及文件**：`src/components/review/ReviewerTaskList.vue`、`src/types/auth.ts`、`src/components/review/reviewPanelActions.ts`
- **依赖**：FE-T2-01；M0 中“`status` 只作展示态、`currentNode` 为流程真源”的约定。
- **完成标准**：
  - `submitted / in_review` 作为 M2 主处理态。
  - 状态徽标、按钮文案、空态文案一致。
  - 筛选项与展示规则有一张明确映射表。
- **风险/备注**：若 `approved / rejected` 继续出现在审核入口主列表，会削弱“待办池”语义。

### FE-T2-03 “开始处理”动作标准化

- **目标**：将 `ReviewerTaskList` 中“开始处理”改为可复用的标准入口动作，不再依赖 `setTimeout` 与隐式面板时序。
- **涉及文件**：`src/components/review/ReviewerTaskList.vue`、`src/composables/useReviewStore.ts`、`src/components/review/reviewPanelActions.ts`
- **依赖**：FE-T2-01；命令总线 `emitCommand` 行为稳定。
- **完成标准**：
  - 进入工作台前先完成当前任务设置。
  - 面板切换不再依赖硬编码延时。
  - “查看详情”和“开始处理”的落点边界清晰。
- **风险/备注**：当前实现依赖异步时序，容易出现任务未设置完成就打开工作台的竞态。

### FE-T2-04 当前任务切换与上下文同步

- **目标**：统一“当前正在处理哪一个任务”的切换逻辑，确保工作台、确认记录、历史和 WS 订阅一起更新。
- **涉及文件**：`src/composables/useReviewStore.ts`、`src/components/review/ReviewerTaskList.vue`、`src/components/review/ReviewPanel.vue`
- **依赖**：FE-T2-03；`reviewStore.setCurrentTask()` 作为唯一任务切换入口。
- **完成标准**：
  - 切换任务时稳定刷新确认记录和历史。
  - 离开任务时能清理旧上下文。
  - 不再出现旧任务数据残留。
- **风险/备注**：`useReviewStore` 目前用 `task.reviewerId` 建立 WS，与 M2 的角色模型不一致，切换任务时可能订阅错人。

### FE-T2-05 提交/打回弹窗入口统一

- **目标**：统一 `WorkflowSubmitDialog`、`WorkflowReturnDialog` 的触发方式和确认回调，替换 `ReviewerTaskList` 内联流转表单。
- **涉及文件**：`src/components/review/ReviewerTaskList.vue`、`src/components/review/WorkflowSubmitDialog.vue`、`src/components/review/WorkflowReturnDialog.vue`
- **依赖**：FE-T2-03、FE-T2-04；`userStore.submitTaskToNextNode()`、`userStore.returnTaskToNode()`。
- **完成标准**：
  - 提交、打回均通过统一弹窗组件触发。
  - 详情区域不再维护独立驳回表单状态。
  - 提交备注/打回原因字段来源一致。
- **风险/备注**：现有 `WorkflowReturnDialog` 支持多级退回，而列表内联逻辑默认“驳回回 sj”，业务规则必须先统一。

### FE-T2-06 节点权限判断收口

- **目标**：将“谁能提交、谁能打回、显示什么按钮”的判断统一收口到共享 helper。
- **涉及文件**：`src/components/review/reviewPanelActions.ts`、`src/components/review/ReviewerTaskList.vue`、`src/components/review/ReviewPanel.vue`
- **依赖**：FE-T2-02、FE-T2-05；节点权限规则确认。
- **完成标准**：
  - 按钮显隐、按钮文案、允许打回目标节点由统一 helper 决定。
  - `ReviewerTaskList` 与工作台判断一致。
- **风险/备注**：当前 helper 只有基础节点能力，没有绑定“当前用户是否正好是该节点处理人”的权限维度。

### FE-T2-07 入口导航与面板落点统一

- **目标**：统一从任务池进入审核工作台的导航路径，明确列表、详情、工作台之间的关系。
- **涉及文件**：`src/components/review/ReviewerTaskList.vue`、`src/components/review/ReviewPanel.vue`、`src/components/review/embedRoleLanding.ts`
- **依赖**：FE-T2-03、FE-T2-04；M2 对导航方案确认。
- **完成标准**：
  - 从待办列表进入工作台只有一条主路径。
  - 查看详情与开始处理的落点边界清晰。
  - embed 场景与普通场景进入结果一致。
- **风险/备注**：当前入口既有列表详情层，也有命令面板切换，还有 embed 落点逻辑，若不统一会继续分叉。

## 4. 后端任务单

### BE-T2-01 审核入口工作流契约冻结

- **目标**：冻结 M2 审核入口依赖的工作流契约，让前端仅依赖 `currentNode + 角色字段 + /submit /return` 即可跑通。
- **依赖**：任务列表接口、工作流接口、角色字段兼容策略。
- **输出**：
  - `GET /api/review/tasks` 对 `checker_id`、`approver_id`、`reviewer_id` 的查询语义
  - `POST /submit` 的节点推进规则
  - `POST /return` 的 `targetNode` 合法集合和状态回写规则
  - `workflow/history` 最小返回字段
- **待确认**：`reviewerId` 是否继续等价映射到 `checkerId`；`/submit` 到 `pz` 是进入批准节点还是直接通过。

### BE-T2-02 旧接口边界确认

- **目标**：明确旧接口 `/start-review`、`/approve`、`/reject` 与新接口 `/submit`、`/return` 的职责边界。
- **依赖**：现有前端状态更新兼容逻辑。
- **输出**：
  - `/start-review` 仅保留历史兼容，M2 新入口默认不再调用
  - `/approve` 仅保留历史兼容或最终批准语义
  - `/reject` 仅保留历史兼容，可内部映射到 `/return`
  - `/submit` 为唯一正向推进接口
  - `/return` 为唯一逆向退回接口
- **待确认**：后端是否接受 M2 前端完全停止主动调用旧接口。

### BE-T2-03 WebSocket 刷新契约补齐

- **目标**：补齐任务池和当前任务视图依赖的最小 WS 事件载荷。
- **依赖**：`/ws/review` 与 `/ws/review/user/{userId}` 通道现状。
- **输出**：事件最小字段：`taskId`、`formId`、`currentNode`、`status`、`operatorId`、`timestamp`；事件类型至少覆盖 `task_updated`、`task_submitted`、`task_returned`。
- **待确认**：继续发送粗粒度事件并让前端全量刷新，还是在 M2 就提供可增量消费 payload。

### BE-T2-04 多级退回规则确认

- **目标**：明确 M2 阶段 `targetNode` 是否允许退回到任意前置节点，还是仅回 `sj`。
- **依赖**：`WorkflowReturnDialog` 当前 UI 能力。
- **输出**：M2 可用退回规则表。
- **待确认**：若后端暂不支持多级退回，前端必须同步降级，不得继续暴露多选退回节点。

## 5. 联调任务单

### INT-T2-01 角色与待办分配联调

- **目标**：验证 checker / approver 与历史 `reviewerId` 兼容场景下，任务不会串人或重复显示。
- **输出**：
  - checker 只看到 `jd` 节点属于自己的任务
  - approver 只看到 `sh/pz` 节点属于自己的任务
  - `reviewerId` 与 `checkerId` 并存时不会重复显示
- **必须确认**：列表接口是否同时返回 `checkerId` 与 `reviewerId`；不一致时谁优先。

### INT-T2-02 开始处理到流转闭环联调

- **目标**：跑通“开始处理 -> 提交/打回 -> 列表刷新 -> 工作流历史刷新”的最短主链路。
- **输出**：
  - checker 节点 `/submit` 后进入下一节点
  - approver 节点 `/return` 后回到指定节点并带回 `returnReason`
  - 列表与工作流历史刷新一致
- **必须确认**：是否还要求 `/start-review` 作为前置动作。

### INT-T2-03 提交/打回弹窗与后端规则联调

- **目标**：保证弹窗里能选/能填的内容，与后端实际支持能力一致。
- **输出**：
  - submit 弹窗仅负责备注，目标节点由后端按 `currentNode` 自动推进
  - return 弹窗可选节点与后端允许值一致
- **必须确认**：M2 是否支持多级退回。

### INT-T2-04 当前任务切换与 WS 联调

- **目标**：验证当前任务切换后，确认记录、历史与 WS 订阅不会留在旧任务上。
- **输出**：
  - 切换任务时上下文刷新顺序
  - 用户专属 WS 应按当前登录用户建链，而非按 `reviewerId`
- **必须确认**：WS 通道绑定字段。

### INT-T2-05 新旧接口边界抓包验证

- **目标**：确认 M2 新入口不再误用 `/approve`、`/reject`、`/start-review`。
- **输出**：
  - 任务列表新入口只打 `/submit`、`/return`
  - 若“开始处理”只是打开工作台，不额外触发旧接口
- **必须确认**：若后端仍强依赖旧接口，此项直接记为 M2 阻塞。

## 6. 验证任务单

### QA-T2-01 审核入口最小闭环验证

- **覆盖**：设计人创建任务后，checker 待办可见；checker 提交后，approver 待办可见；approver 打回后，任务返回指定节点。
- **通过标准**：主链路可跑通，且状态流转一致；审核入口列表、提交弹窗、打回弹窗的视觉与交互以 `review-reviewer.pen` 为验收基准，`review-flow.pen` 仅用于校对跨角色流转关系。

### QA-T2-02 待办筛选与状态口径验证

- **覆盖**：角色切换、节点切换、状态切换、空态场景。
- **通过标准**：筛选结果与按钮权限一致，不出现“可见但不可处理”的错态。

### QA-T2-03 工作流历史留痕验证

- **覆盖**：提交、打回后的历史记录与当前节点回写。
- **通过标准**：`workflow` 可看到对应动作、操作人、时间和备注/原因。

### QA-T2-04 新旧接口边界验证

- **覆盖**：M2 新入口不误用 `/approve`、`/reject`、`/start-review`。
- **通过标准**：抓包或日志中只看到 `/submit`、`/return` 作为主链路请求。

### QA-T2-05 列表刷新与 WS 兜底验证

- **覆盖**：操作后显式刷新、同账号另一端操作后的被动刷新。
- **通过标准**：即使 WS 不稳定，操作后列表仍能通过显式刷新正确更新。

## 7. M2 必须拍板的确认项

以下事项必须在 M2 开发前或开发中期完成确认：

1. `reviewerId` 在 M2 是否继续作为兼容读字段参与待办过滤。
2. `GET /api/review/tasks` 中 `checker_id`、`approver_id`、`reviewer_id` 的正式语义。
3. `/submit` 是否为 M2 唯一正向推进接口。
4. `/return` 是否为 M2 唯一逆向退回接口。
5. `/start-review` 是否仍要求作为前置动作。
6. `targetNode` 在 M2 是否允许退回到任意前置节点。
7. 用户专属 WS 应按当前登录用户还是任务 `reviewerId` 建链。

## 8. 可留到 M3 / M4 的事项
