> 生成时间：2026-03-13  
> 目标：作为 `M0：契约与执行基线` 的落地产物，用于统一页面范围、当前实现、依赖 store、接口动作、字段语义与迁移风险。

> 设计基线：跨角色总流程总览以 `ui/三维校审/review-flow.pen` 为基线，页面归属与角色设计输入以 `ui/三维校审/review-designer.pen`、`ui/三维校审/review-reviewer.pen` 对照

## 1. 使用说明

本文档用于回答 4 个关键问题：

1. 设计稿中的每个页面，在当前仓库里对应到哪个组件或能力块。
2. 每个页面依赖哪些 store、composable、API 和流程动作。
3. 哪些字段/接口是正式主链路，哪些仍属于兼容层。
4. 哪些地方是第一阶段实施的主要风险点。

使用方式建议如下：

- 前端开发：先查第 2 章页面映射表，确认目标页面与当前代码落点。
- 后端开发：先查第 3-7 章字段与接口映射，确认兼容窗口和迁移目标。
- 联调测试：结合第 8 章风险清单，优先覆盖高风险链路。
- 涉及设计输入时：发起人页面优先对照 `review-designer.pen`，审核侧页面优先对照 `review-reviewer.pen`，跨角色流程关系再回看 `review-flow.pen`。

## 3. 核心任务字段映射

| 当前字段 | 当前位置 | 目标语义 | 兼容策略 | 风险 |
| --- | --- | --- | --- | --- |
| `ReviewTask.id` | `src/types/auth.ts`、`src/api/reviewApi.ts` | 内部任务主键 `taskId` | 保持不变 | 若误当作外部单据号使用，会与 `formId` 混用 |
| `ReviewTask.formId` / `form_id` | `src/types/auth.ts`、`src/api/reviewApi.ts` | 外部系统业务单号 | 前端统一使用 `formId`，解析时兼容 `form_id` | 旧数据不返回 `formId` 时，embed 与 aux-data 关联会断 |
| `title` | `src/api/reviewApi.ts` | 任务/单据标题 | 直接透传 | 外部系统命名若不同，需文档统一 |
| `description` | `src/api/reviewApi.ts` | 编校审说明/说明 | 直接透传，可空 | 空字符串与 `undefined` 语义不统一 |
| `modelName` / `model_name` | `src/api/reviewApi.ts` | 模型名称 | 兼容 snake/camel | 名称不可作为模型主键 |
| `status` | `src/types/auth.ts`、`src/api/reviewApi.ts` | 生命周期状态 | 限定枚举，非法值兜底 `draft` | 后端新增状态时前端会失真 |
| `priority` | `src/types/auth.ts`、`src/api/reviewApi.ts` | 优先级 | 限定枚举，非法值兜底 `medium` | 新增优先级时前端无法识别 |
| `requesterId` / `requesterName` | `src/types/auth.ts`、`src/api/reviewApi.ts` | 发起人 | 兼容 snake/camel | embed 当前用户与发起人容易混淆 |
| `checkerId` / `checkerName` | `src/types/auth.ts`、`src/api/reviewApi.ts` | 第二节点校对人员 | 作为正式字段保留 | 旧接口只识别 `reviewerId` 时会产生歧义 |
| `approverId` / `approverName` | `src/types/auth.ts`、`src/api/reviewApi.ts` | 第三节点审核人 | 作为正式字段保留 | 若后续拆出批准节点责任人，当前结构不够细 |
| `reviewerId` / `reviewerName` | `src/types/auth.ts`、`src/api/reviewApi.ts`、`src/composables/useUserStore.ts` | 旧 reviewer 语义，现兼容映射到 `checker` | 兼容期双读；创建时允许 `reviewerId=checkerId` | 在 `useReviewStore` 中仍用作 WS 建链，存在角色错配风险 |
| `components[].refNo` | `src/types/auth.ts`、`ReviewAuxData.vue` | 模型构件编码 | 前端统一 `refNo`；请求辅助数据时映射成 `model_refnos[]` | 若缺少 `refNo`，辅助数据无法查询 |
| `attachments[]` | `src/types/auth.ts`、`src/api/reviewApi.ts` | 任务附件 | 解析兼容 `file_id/file_name/download_url/file_size` | 外部附件元数据字段不统一 |
| `currentNode` | `src/types/auth.ts`、`src/api/reviewApi.ts` | 当前流程节点：`sj/jd/sh/pz` | 统一为四段节点码 | 外部扩展节点时前端会被压扁 |
| `workflowHistory` | `src/types/auth.ts`、`src/api/reviewApi.ts` | 流转历史 | 兼容 snake/camel | 若字段不全，时间线信息会不完整 |
| `returnReason` | `src/types/auth.ts`、`src/api/reviewApi.ts` | 退回原因 | 兼容 snake/camel | 退回原因来源不统一会影响返工大盘 |

## 4. 流程动作接口映射

| 当前接口 | 当前位置 | 目标语义 | 兼容策略 | 风险 |
| --- | --- | --- | --- | --- |
| `POST /api/review/tasks` | `src/api/reviewApi.ts` | 创建任务/编校审单 | 请求体同时传 `checkerId/approverId/reviewerId/formId`，兼容新旧后端 | 若后端只消费一组角色字段，会导致责任人错配 |
| `GET /api/review/tasks` | `src/api/reviewApi.ts` | 查询任务列表 | 过滤同时支持 `checkerId/approverId/reviewerId` | reviewer 与 checker 若被后端视作不同语义，会造成列表偏差 |
| `GET /api/review/tasks/{taskId}` | `src/api/reviewApi.ts` | 查询任务详情 | 返回体统一经 `normalizeReviewTask()` 标准化 | 原始字段缺失时前端静默兜底，容易掩盖数据问题 |
| `PATCH /api/review/tasks/{taskId}` | `src/api/reviewApi.ts` | 更新任务基础信息/附件 | 仅允许编辑基础字段，不负责流转 | 若被滥用于流程推进，会破坏状态机 |
| `DELETE /api/review/tasks/{taskId}` | `src/api/reviewApi.ts` | 删除任务 | 直接调用 | 若后端仅允许逻辑删除，需另行定义 |
| `POST /api/review/tasks/{taskId}/start-review` | `src/api/reviewApi.ts` | 开始处理/进入处理中 | 兼容期保留 | 多级流程下不区分校对开始与审核开始 |
| `POST /api/review/tasks/{taskId}/approve` | `src/api/reviewApi.ts` | 通过当前环节或整单通过 | 兼容期保留；建议未来仅用于最终批准 | “当前环节通过”与“整单通过”语义不清 |
| `POST /api/review/tasks/{taskId}/reject` | `src/api/reviewApi.ts` | 驳回 | 兼容期保留；后续可视作 `/return` 别名 | 与 `/return` 功能重叠 |
| `POST /api/review/tasks/{taskId}/cancel` | `src/api/reviewApi.ts` | 取消任务 | 保留 | `reason/comment` 命名不统一 |
| `POST /api/review/tasks/{taskId}/submit` | `src/api/reviewApi.ts` | 确认流转到下一节点 | 建议作为正式标准流转动作 | 若后端不按 `currentNode` 自动推进，会卡流程 |
| `POST /api/review/tasks/{taskId}/return` | `src/api/reviewApi.ts` | 确认驳回流转到指定节点 | 建议作为正式标准退回动作 | `targetNode` 仅支持四段节点 |
| `GET /api/review/tasks/{taskId}/workflow` | `src/api/reviewApi.ts` | 查询工作流当前节点与历史 | 保留 | `currentNodeName` 只能用于展示，不宜程序判断 |
| `GET /api/review/tasks/{taskId}/history` | `src/api/reviewApi.ts` | 查询审核动作日志 | 保留，作为审计日志 | 与 workflow history 两套口径并存，易重复 |

| `embed_landing_state` | `src/components/review/embedRoleLanding.ts` | embed 初始面板落点 | 缓存 `{ target, formId, primaryPanelId, visiblePanelIds }` | 面板 ID 改名会导致落点失效 |
| `resolveEmbedLandingTarget()` | `src/components/review/embedRoleLanding.ts` | 根据 embed 与角色决定落点 | 当前仅看 `isDesigner/isReviewer` | 无法仅靠参数强制区分 checker/approver |
| `POST /api/review/aux-data` | `src/api/reviewApi.ts`、`ReviewAuxData.vue` | 获取外部辅助校审数据 | 请求体 `{ project_id, model_refnos, major, requester_id, page, page_size, form_id, new_search }`，鉴权走 `UCode/UKey` | 与主 review token 体系割裂 |
| `model_refnos[]` | `src/api/reviewApi.ts`、`ReviewAuxData.vue` | 辅助数据查询构件编码集合 | 由 `currentTask.components[].refNo` 映射 | 缺少 `refNo` 时会空查 |
| `major` | `ReviewAuxData.vue`、`src/api/reviewApi.ts` | 专业编码 | 当前默认 `general` | 若外部不是自由文本，默认值可能无效 |
| `requester_id` | `ReviewAuxData.vue`、`src/api/reviewApi.ts` | 发起查询的当前用户 ID | 当前取 `currentUser.id`，兜底 `guest` | 与任务发起人 `requesterId` 语义不同 |
| `form_id`（aux-data） | `ReviewAuxData.vue`、`src/api/reviewApi.ts` | 辅助数据关联单据号 | 当前优先输入框，否则回落 `currentTask.id` | 正式业务逻辑中回落 `task.id` 是明显风险 |
| `UCode` / `UKey` Header | `src/api/reviewApi.ts`、`ReviewAuxData.vue` | 辅助数据鉴权 | 当前本地保存 `review_aux_ucode/review_aux_ukey` | 明文 localStorage 存凭证，有安全风险 |

## 8. 主要风险与 M0 输出要求

### 8.1 风险清单

- `reviewerId` 语义仍残留在任务创建、兼容解析和 WS 建链中，是最主要的历史包袱。
- `status` 与 `currentNode` 双真源共存，若没有统一口径，页面筛选与权限判断会继续分叉。
- `ReviewAuxData` 仍沿用调试式参数输入和本地明文鉴权，不适合作为正式产品链路。
- 批注评论、确认记录、流转历史虽然已分层，但当前 UI 上的入口与展示还没有完成完全解耦。
- embed 落点仍依赖本地角色判断和 `sessionStorage`，不够标准化。

### 8.2 M0 必须产出的确认项

- 确认正式字段：`taskId`、`formId`、`checkerId`、`approverId`、`currentNode`、`workflowHistory`。
- 确认兼容字段与废弃窗口：`reviewerId/reviewerName`、`/start-review`、`/approve`、`/reject`。
- 确认正式流转动作：优先 `/submit`、`/return`。
- 确认 embed 正式参数：至少 `form_id`、`project_id`、`user_id`、角色落点字段。
- 确认辅助数据正式鉴权方式与 `form_id` 来源。
