# 三维校审 M4 审核工作台基础版任务单

> 生成时间：2026-03-13  
> 适用阶段：`M4：审核工作台基础版`  
> 对应范围：`ReviewPanel`、`ReviewAuxData`  
> 主设计源：`ui/三维校审/review-reviewer.pen`  
> 总览基线：`ui/三维校审/review-flow.pen`（仅用于跨角色流程与页面衔接对照）  
> 配套文档：`开发文档/三维校审/三维校审重构执行方案.md`、`开发文档/三维校审/三维校审M0页面接口字段映射表.md`

## 1. M4 目标

M4 的目标是让审核工作台从“堆叠式功能集合”升级为有清晰分区和稳定上下文的正式工作台。M4 必须完成以下 5 件事：

1. `ReviewPanel` 形成稳定的信息架构与分区。
2. 当前任务上下文展示口径统一。
3. 流转动作、确认记录与工作台主界面整合。
4. `ReviewAuxData` 从调试面板升级为工作台正式子区。
5. 辅助数据、碰撞、同步、工作流刷新可以在任务上下文内跑通。

## 3. 前端任务单

### FE-T4-01 审核工作台布局重构

- **目标**：将 `ReviewPanel` 重构为固定分区的正式工作台。
- **涉及文件**：`src/components/review/ReviewPanel.vue`
- **依赖**：`useReviewStore`、`useUserStore`、M4 信息架构确认。
- **完成标准**：
  - 至少区分任务上下文区、流转区、确认记录区、辅助数据区、同步区。
  - 不再混放重复查询状态和临时调试逻辑。
- **风险/备注**：是否保留附件区和 embed reviewer 提示，需要产品/设计确认。

### FE-T4-02 当前任务上下文标准化展示

- **目标**：统一工作台对 `title/modelName/requester/checker/approver/currentNode/formId/components` 的展示口径。
- **涉及文件**：`src/components/review/ReviewPanel.vue`、`src/api/reviewApi.ts`
- **依赖**：`normalizeReviewTask()`；M0 字段映射结论。
- **完成标准**：
  - 正式优先显示 `checker/approver/formId/currentNode`。
  - `reviewerName` 仅作为兼容兜底，不作主渲染字段。
- **风险/备注**：历史数据若缺字段，兜底策略需明确。

### FE-T4-03 流转动作与工作台整合

- **目标**：将提交/打回动作整合成工作台内标准流转区。
- **涉及文件**：`src/components/review/ReviewPanel.vue`、`src/components/review/reviewPanelActions.ts`
- **依赖**：`submitTaskToNextNode()`、`returnTaskToNode()`、`getTaskWorkflowHistory()`。
- **完成标准**：
  - 工作台流转只认标准动作入口。
  - 按钮文案与当前节点一致。
  - 动作成功后工作流历史刷新。
- **风险/备注**：M4 是否允许“驳回仅回 sj”写死，必须先确认。

### FE-T4-04 确认记录区基础整合

- **目标**：把待确认数据、已确认统计、确认历史整合进工作台，但与流转历史保持清晰边界。
- **涉及文件**：`src/components/review/ReviewPanel.vue`、`src/composables/useReviewStore.ts`
- **依赖**：`useToolStore`、确认记录接口。
- **完成标准**：
  - 确认动作可用。
  - 确认历史可查看与删除。
  - 统计值与记录列表一致。
- **风险/备注**：删除/清空是否暴露在 M4，需要权限口径确认。

### FE-T4-05 `ReviewAuxData` 产品化

- **目标**：把 `ReviewAuxData` 从调试式表单升级为基于任务上下文的正式工作台子区。
- **涉及文件**：`src/components/review/ReviewAuxData.vue`、`src/components/review/ReviewPanel.vue`
- **依赖**：`reviewGetCollisionData()`、`reviewGetAuxData()`、`currentTask`、`currentUser`。
- **完成标准**：
  - 用户无需手输核心参数即可发起查询。
  - 碰撞结果支持定位/高亮。
  - `formId` 正式来自任务上下文，而不是回落 `task.id`。
- **风险/备注**：`UCode/UKey` 是否仍需临时暴露，必须拍板。

### FE-T4-06 同步区并入工作台语义

- **目标**：将导入/导出同步能力收口为工作台基础区块。
- **涉及文件**：`src/components/review/ReviewPanel.vue`、`src/components/review/ReviewDataSync.vue`
- **依赖**：`reviewSyncExport()`、`reviewSyncImport()`。
- **完成标准**：
  - 用户理解导入/导出作用范围。
  - 导入后当前任务上下文可刷新。
- **风险/备注**：导入覆盖语义不清时，必须加风险提示。

## 4. 后端任务单

### BE-T4-01 工作台上下文字段口径确认

- **目标**：确认工作台依赖的正式字段集合。
- **依赖**：任务模型与 `normalizeReviewTask()` 现状。
- **输出**：`taskId/formId/checker/approver/currentNode/workflowHistory/returnReason` 的正式字段清单。
- **待确认**：`formId` 缺失时是否允许为空而不是回落 `task.id`。

### BE-T4-02 流转接口语义收口

- **目标**：确认工作台正式只用 `/submit`、`/return`。
- **依赖**：任务流转接口、workflow 接口。
- **输出**：节点推进规则、退回规则、错误语义说明。
- **待确认**：是否仍需 `/start-review` 前置；批准节点是否单独走 `/approve`。

### BE-T4-03 辅助数据正式入参确认

- **目标**：冻结 `ReviewAuxData` 的正式请求参数来源。
- **依赖**：`POST /api/review/aux-data` 现有契约。
- **输出**：`project_id/model_refnos/major/requester_id/form_id` 的必填性与来源说明。
- **待确认**：短期是否继续沿用 `UCode/UKey`。

### BE-T4-04 确认记录边界确认

- **目标**：确认记录接口只承载确认快照，不与 workflow history 或评论混写。
- **依赖**：记录接口与 workflow/comment 接口边界。
- **输出**：记录查询、删除、清空的权限与语义说明。
- **待确认**：是否补充 `formId` 维度查询。

### BE-T4-05 最小实时事件集确认

- **目标**：确认 M4 需要的最小 WS 事件集。
- **依赖**：全局任务 WS 与用户专属 WS。
- **输出**：至少覆盖任务变更、确认记录保存的事件集与最小载荷。
- **待确认**：WS 用户通道最终绑定当前登录用户还是历史 `reviewerId`。

## 5. 联调任务单

### INT-T4-01 工作台主链路联调

- **目标**：联调“进入工作台 -> 查看上下文 -> 查询辅助数据/碰撞 -> 生成确认记录 -> 提交/打回 -> 刷新历史”闭环。
- **输出**：至少一条真实任务的全链路联调记录。
- **必须确认**：`formId/project_id/refNo` 任一缺失的处理策略。

### INT-T4-02 辅助数据与 viewer 联动联调

- **目标**：验证碰撞结果定位/高亮与任务上下文一致。
- **输出**：定位、高亮、任务切换后查询上下文刷新正常。
- **必须确认**：viewer objectId 与业务 `refNo` 的映射稳定性。

### INT-T4-03 同步区联调

- **目标**：验证导入/导出不会破坏当前任务上下文。
- **输出**：导入导出成功/失败样例与刷新结果。
- **必须确认**：导入覆盖语义。

## 6. 验证任务单

### QA-T4-01 工作台基础能力验收

- **覆盖**：任务上下文、流转动作、确认记录、碰撞查询、辅助数据、同步区。
- **通过标准**：六大块均能在任务上下文内独立使用；审核工作台页面与辅助数据子区的视觉、信息架构和交互以 `review-reviewer.pen` 为验收基准，`review-flow.pen` 仅用于校对跨角色流转与入口衔接。

### QA-T4-02 边界与回归验证

- **覆盖**：M4 不误伤 M5/M6 边界能力。
- **通过标准**：批注入口仍可用，但不要求新悬浮工具栏；评论面板未纳入主链路时不影响 M4 闭环。

## 7. M4 必须拍板的确认项

1. `ReviewPanel` 的正式分区结构。
2. 当前任务上下文的正式字段口径。
3. 流转入口统一走 `/submit`、`/return`。
4. 确认记录、流转历史、评论三条数据线严格分治。
5. `ReviewAuxData` 的正式入参来源，尤其 `form_id`。
6. 最小实时刷新事件集和用户通道语义。

## 8. 可留到 M5 / M6 的事项

- 批注工具栏与批注列表正式拆分
- 悬浮确认交互重设计
- 角色分栏评论协同与实时编辑/删除同步
- 更复杂的辅助数据可视化
