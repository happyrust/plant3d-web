# 三维校审 M4 审核工作台基础版会议 Checklist

> 用途：开会快速对齐 + 会后直接派单  
> 主设计源：`ui/三维校审/review-reviewer.pen`  
> 总览基线：`ui/三维校审/review-flow.pen`（仅用于跨角色流程与入口衔接对照）  
> 对应文档：`开发文档/三维校审/三维校审M4审核工作台基础版任务单.md`

## 一、会议目标

- [ ] 对齐 M4 只做“审核工作台基础版”，不扩到 M5 / M6
- [ ] 拍板工作台分区、上下文字段、流转与辅助数据边界
- [ ] 明确确认记录、流转历史、评论三条线的边界
- [ ] 会后可以直接按 owner 派单开工

## 二、会前必读

- [ ] 阅读 `开发文档/三维校审/三维校审M4审核工作台基础版任务单.md`
- [ ] 阅读 `开发文档/三维校审/三维校审重构执行方案.md`
- [ ] 阅读 `开发文档/三维校审/三维校审M0页面接口字段映射表.md`
- [ ] 对照 `ui/三维校审/review-reviewer.pen` 确认审核工作台与辅助数据主设计
- [ ] 如需核对跨角色流程和入口衔接，再参考 `ui/三维校审/review-flow.pen`
- [ ] 前端确认范围页面：`ReviewPanel` / `ReviewAuxData`
- [ ] 后端准备任务上下文字段、submit/return、aux-data、records、WS 现状

## 三、必须拍板项

- [ ] `ReviewPanel` 的正式分区结构
- [ ] 当前任务上下文的正式字段口径
- [ ] 流转动作是否统一只走 `/submit`、`/return`
- [ ] 确认记录、流转历史、评论是否严格分治
- [ ] `ReviewAuxData` 的正式入参来源，尤其 `form_id`
- [ ] 最小实时刷新事件集及用户通道语义

## 四、前端 Owner 清单

- [ ] FE1：重构 `ReviewPanel` 为固定分区工作台
- [ ] FE2：标准化任务上下文展示，优先正式字段
- [ ] FE3：将提交/打回动作整合进统一流转区
- [ ] FE4：整合确认记录区，明确统计、历史、删除边界
- [ ] FE5：将 `ReviewAuxData` 产品化，避免用户手填核心参数
- [ ] FE6：将同步区并入工作台语义并支持导入后刷新

## 五、后端 Owner 清单

- [ ] BE1：冻结工作台正式字段集合与缺失兜底策略
- [ ] BE2：收口流转接口语义，说明推进、退回、错误规则
- [ ] BE3：明确辅助数据接口正式参数来源及必填性
- [ ] BE4：确认确认记录接口与 workflow/comment 的边界
- [ ] BE5：确认最小 WS 事件集与载荷范围

## 六、联调 / 测试清单

- [ ] INT1：联调“进入工作台 -> 辅助数据/碰撞 -> 确认记录 -> submit/return -> 历史刷新”主链路
- [ ] INT2：验证碰撞结果与 viewer 定位/高亮联动稳定
- [ ] INT3：验证任务切换后辅助数据查询上下文同步刷新
- [ ] INT4：验证导入/导出同步不破坏当前任务上下文
- [ ] QA1：覆盖上下文、流转、确认记录、辅助数据、同步六大块
- [ ] QA1-视觉基线：`ReviewPanel`、`ReviewAuxData` 按 `review-reviewer.pen` 验收
- [ ] QA2：回归确认 M4 不误伤 M5 批注入口和 M6 评论边界

## 七、风险提醒

- [ ] 风险：`formId` 若继续回落 `task.id`，后续上下文会持续混乱
- [ ] 风险：流转、确认、评论边界不清，工作台会再次长成超级面板
- [ ] 风险：辅助数据若仍依赖临时参数，产品化会失败
- [ ] 风险：WS 事件定义不清，会导致刷新时机和用户归属混乱
