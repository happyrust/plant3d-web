### 为什么选 Phase C

1. **DUAL_READ 基础已就位**：`commentThreadDualRead.*` / `commentEventLog.*` / `sharedStores.*` 已合入 main
2. **直接解决 P1 根因**：评论塞在 `annotation.comments[]` 内是 P1 本地 ID 不一致问题的结构性根因
3. **M6 主线任务**：与路线图 M6（评论协同）直接对齐

### 开发任务分解

#### C1: commentThreadStore 独立真源 ✅ 已完成

已有完整实现：
- `src/review/services/commentThreadStore.ts` — CRUD + subscribe + fingerprint 去重
- `src/review/services/commentThreadDualRead.ts` — inline ↔ store 双源同步 + diff
- `src/review/services/commentEventLog.ts` — 环形 buffer 诊断信道（500 条）
- `src/review/services/sharedStores.ts` — 全局单例 + feature flag 联动
- `src/review/flags.ts` — `REVIEW_C_COMMENT_THREAD_STORE_DUAL_READ` / `CUTOVER` 开关
- 所有 flag 默认 `false`，通过 `localStorage` 或 `VITE_*` 环境变量启用

#### C2: AnnotationPanel 评论读写迁移（0.5 天）

- `AnnotationPanel.vue` 的评论操作（`addComment` / `saveEditComment` / `deleteComment` / `loadCommentsForListView`）迁移到 `commentThreadStore`
- 移除组件内直接调用 `reviewApi` 评论接口
- `activeComments` computed 改为从 `commentThreadStore` 读取

#### C3: ReviewCommentsTimeline / ReviewCommentsPanel 迁移（0.5 天）

- 这两个组件的评论加载和展示也迁移到 `commentThreadStore`
- 统一加载逻辑，避免多组件各自拉取

#### C4: useToolStore.comments 降级为投影（0.5 天）

- `addCommentToAnnotation` / `setAnnotationComments` / `updateAnnotationComment` / `removeAnnotationComment` 改为仅更新本地投影
- 标记为 `@deprecated`，新代码不再直接调用
- 确认快照 `buildReviewConfirmSnapshotPayload` 中的 comments 字段来源也切换

#### C5: DUAL_READ 观察与差异审计（持续）

- 利用已有的 `commentEventLog` 记录差异
- 观察 DUAL_READ 期间 inline 真源 vs store 真源的偏差
- 达标后进入 PROMOTE 阶段

### 验收标准

- [ ] 评论 CRUD 全部通过 `commentThreadStore`，组件不直接调 API
- [ ] `useToolStore.annotation.comments` 仍能渲染（兼容投影）
- [ ] 回放恢复后评论正确展示
- [ ] P1 场景彻底消失：无论 API 成功或失败，不会出现重复评论
- [ ] `commentEventLog` 无 `dual_read_diff` 事件（表明两个真源一致）

### 风险

| 风险 | 应对 |
|---|---|
| `useToolStore.comments` 有其他隐式消费者 | 全局搜索 `annotation.comments` 确认消费点 |
| 确认快照包含 comments 字段的序列化格式变化 | 兼容期仍从 `useToolStore` 投影读取 |
| 嵌入恢复链路的 comments 来源不同 | `embedFormSnapshotRestore` 也走 `commentThreadStore` |
