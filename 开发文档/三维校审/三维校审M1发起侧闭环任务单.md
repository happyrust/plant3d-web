# 三维校审 M1 发起侧闭环任务单

> 生成时间：2026-03-13  
> 适用阶段：`M1：发起侧闭环`  
> 对应范围：`InitiateReviewPanel`、`FileUploadSection`、`AssociatedFilesList`  
> 主设计源：`ui/三维校审/review-designer.pen`  
> 跨角色总览基线：`ui/三维校审/review-flow.pen`（仅用于核对整体流转关系）  
> 配套文档：`开发文档/三维校审/三维校审重构执行方案.md`、`开发文档/三维校审/三维校审M0页面接口字段映射表.md`

## 1. M1 目标

M1 的目标不是一次性完成全部三维校审流程，而是先稳定发起侧的正式业务闭环。必须完成以下 5 件事：

1. 设计/编制人员能稳定创建提资单。
2. 创建成功后能拿到可靠的 `taskId` 与 `formId`。
3. 附件能按统一 lineage 归档、回写并在任务中可见。
4. 关联文件区域不再使用纯 mock，而是具备真实业务占位能力。
5. embed 进入发起页时，`form_id/project_id/user_id/user_token` 的来源与透传规则明确。

## 3. 前端任务单

### FE-T1 发起面板数据模型收口

- **目标**：将 `InitiateReviewPanel` 的表单、构件、embed 上下文、附件状态统一到一份明确的数据模型中。
- **涉及文件**：`src/components/review/InitiateReviewPanel.vue`、`src/composables/useUserStore.ts`、`src/api/reviewApi.ts`
- **依赖**：M0 字段映射结论；后端确认 `taskId` / `formId` 返回口径。
- **完成标准**：
  - 创建提资单的请求字段来源清晰，区分用户输入、嵌入透传、构件选择、附件回写。
  - `formId` 只允许来自“embed 透传”或“后端回写”。
  - 页面内不再通过构件名称或临时默认值推导业务主键。
- **风险/备注**：当前 `InitiateReviewPanel` 同时承担构件、附件、embed、viewer 打开，职责过重，是 M1 最先需要解耦的页面。

### FE-T2 提资创建链路标准化

- **目标**：收口“创建任务 -> 回写 `taskId/formId` -> 刷新页面状态”的单一路径。
- **涉及文件**：`src/components/review/InitiateReviewPanel.vue`、`src/composables/useUserStore.ts`
- **依赖**：FE-T1；`reviewTaskCreate` 返回体稳定包含 `id`，优先包含 `formId`。
- **完成标准**：
  - 创建成功后页面进入明确的“已创建”状态。
  - `createdTaskId`、`createdTaskFormId` 的写入时机唯一。
  - 创建失败时不会残留半初始化状态。
- **风险/备注**：当前创建后立即串附件上传和附件回写，若 `formId` 缺失会直接影响后续附件归档。

### FE-T3 附件上传状态机拆分

- **目标**：把 `FileUploadSection` 升级为正式附件状态组件，明确 `pending / uploading / success / error` 四类状态。
- **涉及文件**：`src/components/review/FileUploadSection.vue`、`src/components/review/InitiateReviewPanel.vue`、`src/components/review/reviewAttachmentFlow.*`
- **依赖**：FE-T2；附件接口对 `taskId/formId` 归档规则稳定。
- **完成标准**：
  - 创建前附件只处于待上传缓存态。
  - 创建后按统一时机触发自动上传。
  - 失败文件可重试，且不影响其他文件。
  - 父组件只提供 lineage，不直接耦合上传细节。
- **风险/备注**：当前 `autoUpload`、`startUpload`、`hasAttachmentLineage` 三套逻辑交织，是 M1 最容易返工的点之一。

### FE-T4 附件元数据回写与任务附件同步

- **目标**：统一附件上传成功后的元数据回写机制，保证任务详情中的 `attachments` 来源唯一。
- **涉及文件**：`src/components/review/InitiateReviewPanel.vue`、`src/components/review/reviewAttachmentFlow.*`、`src/composables/useUserStore.ts`
- **依赖**：FE-T3；`PATCH /api/review/tasks/{taskId}` 的附件更新语义稳定。
- **完成标准**：
  - 上传成功后，任务附件列表由统一 builder 生成。
  - `updateTaskAttachments` 调用时机固定且不会重复写入。
  - 上传失败文件不会误进入任务附件。
- **风险/备注**：当前创建提交与上传完成都可能触发附件同步，存在重复回写窗口。

### FE-T5 关联文件区升级为真实业务占位

- **目标**：把 `AssociatedFilesList` 从静态 mock 改成可承接真实接口的业务占位组件。
- **涉及文件**：`src/components/review/AssociatedFilesList.vue`、`src/components/review/InitiateReviewPanel.vue`
- **依赖**：FE-T1；后端确认关联文件查询维度。
- **完成标准**：
  - 组件不再内置固定假数据。
  - 支持 `loading / empty / error / data` 四类状态。
  - 文件点击动作统一为事件协议，而不是 `console.log`。
- **风险/备注**：如果后端在 M1 无法提供真实接口，则本任务至少要把组件改造成“真实占位”而不是继续假数据。

### FE-T6 构件选择与关联上下文整理

- **目标**：统一“选中构件 -> 拉取属性 -> 生成 `ReviewComponent` -> 作为创建任务和关联文件输入”的链路。
- **涉及文件**：`src/components/review/InitiateReviewPanel.vue`、`src/composables/useSelectionStore.ts`
- **依赖**：FE-T1；三维选择上下文稳定提供 `refNo`。
- **完成标准**：
  - 构件对象结构统一，至少明确 `id/name/refNo/type`。
  - 属性拉取失败时的兜底规则明确。
  - `selectedComponents` 可直接用于创建任务和关联文件查询。
- **风险/备注**：若 `refNo` 不稳定，M1 后续的附件按构件归档和关联文件按构件查询都会出错。

### FE-T7 发起侧校验与反馈统一

- **目标**：统一 M1 页面中的校验、通知与异常反馈，形成可测试的用户闭环。
- **涉及文件**：`src/components/review/InitiateReviewPanel.vue`、`src/components/review/FileUploadSection.vue`
- **依赖**：FE-T2、FE-T3。
- **完成标准**：
  - 必填项、人员冲突、附件失败、创建失败、部分附件失败都有明确提示。
  - 创建成功提示能准确反映“提资单创建结果 + 附件结果”。
  - 无效状态下按钮禁用规则稳定。
- **风险/备注**：不统一反馈口径，联调后将很难沉淀稳定验收用例。

## 4. 后端任务单

### BE-T1 创建任务接口收口

- **目标**：明确 `POST /api/review/tasks` 的正式入参/回参。
- **依赖**：前端当前创建链路已传 `checkerId`、`approverId`、兼容 `reviewerId`、可选 `formId`。
- **输出**：接口契约或联调样例；返回体稳定包含 `id`、`formId`、`checkerId`、`approverId`、`currentNode`、`status`。
- **待确认**：
  - `reviewerId` 是否仍要求必传。
  - 是否允许沿用外部已创建单据的 `formId`。

### BE-T2 附件上传归档规则补齐

- **目标**：明确 `POST /api/review/attachments` 在仅有 `taskId`、仅有 `formId`、二者同时存在时的归档规则。
- **依赖**：前端上传已支持传 `taskId` 与 `formId`。
- **输出**：附件上传归档说明；成功回参包含标准附件元数据。
- **待确认**：
  - 创建前是否允许仅凭 `formId` 先传附件。
  - 是否支持创建后自动挂接到 `taskId`。

### BE-T3 任务附件回写口径统一

- **目标**：统一任务详情和任务更新接口中的 `attachments` 结构。
- **依赖**：前端通过 `PATCH /api/review/tasks/{taskId}` 回写附件。
- **输出**：附件字段说明；列表/详情响应中的标准附件结构。
- **待确认**：`attachments` 以后端存储记录为准，还是允许前端整体覆盖。

### BE-T4 关联文件查询接口落地

- **目标**：提供 `AssociatedFilesList` 的真实数据源。
- **依赖**：需要明确关联文件来源系统与查询维度。
- **输出**：按 `taskId` 或 `formId + refno[]` 查询的接口；返回分类、状态、更新时间、下载地址。
- **待确认**：M1 是否必须接入真实接口；若不能，需明确降级方案。

### BE-T5 embed 参数口径确认

- **目标**：确认 M1 所需的最小 embed 参数集。
- **依赖**：当前前端通过 `reviewGetEmbedUrl()` 组装 query，并消费 `embed_mode_params`。
- **输出**：正式参数表与回包样例，至少覆盖 `form_id`、`project_id`、`user_id`、`user_token`。
- **待确认**：`landing_role` 是否 M1 就要支持；`token` 是否允许放在 query。

### BE-T6 下载/预览权限策略

- **目标**：定义上传附件和关联文件的下载/预览地址与权限策略。
- **依赖**：前端当前只消费 `attachment.url`，关联文件尚无真实打开逻辑。
- **输出**：下载 URL 规则、鉴权说明、跨系统跳转说明。
- **待确认**：embed 场景是否免登；是否需要短链或临时签名。

## 5. 联调任务单

### INT-T1 任务创建回写联调

- **目标**：验证创建任务后能稳定回写 `taskId` 与 `formId`。
- **输出**：成功/失败样例；字段缺失时的处理约定。
- **必须确认**：若后端不返回 `formId`，M1 是否允许继续。建议不允许静默缺失。

### INT-T2 附件上传时序联调

- **目标**：验证“先创建任务，再自动上传附件，再回写任务附件”的完整时序。
- **输出**：时序图或抓包记录；覆盖无附件、全部成功、部分失败三类场景。
- **必须确认**：上传成功后是否仍必须显式 `PATCH tasks/{id}`。

### INT-T3 `taskId/formId` 双主键联调

- **目标**：验证创建、上传、查询、关联文件、embed 打开时，技术主键与业务单号不混用。
- **输出**：字段使用矩阵。
- **必须确认**：哪些接口只认 `taskId`，哪些接口只认 `formId`，哪些兼容二者。

### INT-T4 关联文件真实数据联调

- **目标**：将 `AssociatedFilesList` 从 mock 切换到真实数据。
- **输出**：至少 1 个 task/formId 的真实文件样例；分类、状态、打开动作可验证。
- **必须确认**：若 M1 验收包含“关联文件展示”，本项必须完成。

### INT-T5 embed 参数透传联调

- **目标**：验证 embed 进入发起页时，`form_id/project_id/user_id/user_token` 被正确读取与透传。
- **输出**：参数样例、前端读取结果、外部页面落点结果。
- **必须确认**：参数来源究竟是 URL、sessionStorage 还是后端接口回写。

### INT-T6 角色字段兼容联调

- **目标**：确认后端是否仍依赖 `reviewerId`，避免发起成功后审核分配错误。
- **输出**：兼容说明；`reviewerId` 的下线前提。
- **必须确认**：M1 内必须拍板，否则 M2 会重复返工。

## 6. 验证任务单

### QA-T1 发起创建验证

- **覆盖**：正常创建、缺校核人、缺审核人、校核人与审核人相同。
- **通过标准**：能稳定返回 `taskId` 与 `formId`；错误提示准确。

### QA-T2 附件链路验证

- **覆盖**：单文件、多文件、非法类型、超大文件、部分失败重试。
- **通过标准**：上传结果与任务附件回写一致；失败文件不污染成功结果。

### QA-T3 主键一致性验证

- **覆盖**：创建、上传、详情查询、embed 打开、关联文件查询。
- **通过标准**：`taskId/formId` 使用口径一致，无隐式替换或错误回退。

### QA-T4 关联文件展示验证

- **覆盖**：空结果、有结果、分类展示、状态标签、点击打开。
- **通过标准**：若 M1 包含真实关联文件，则必须使用真实数据通过；否则记为阻塞项，不得伪通过。

### QA-T5 embed 场景验证

- **覆盖**：带 `formId`、不带 `formId`、缺 `projectId` 三类场景。
- **通过标准**：发起页上下文正确；外部校审入口可打开；不依赖测试兜底值。

### QA-T6 最小回归验证

- **覆盖**：创建后任务列表可见、任务详情可查、附件字段可解析。
- **通过标准**：M1 不破坏现有创建任务与读取能力。

## 7. M1 必须拍板的确认项

以下事项必须在 M1 开发前或开发中期完成确认：

1. `POST /api/review/tasks` 回包必须稳定返回 `id` 与 `formId`。
2. `POST /api/review/attachments` 对 `taskId/formId` 的归档规则必须明确。
3. 任务附件回写与任务详情中的 `attachments` 字段口径必须一致。
4. `AssociatedFilesList` 是否在 M1 提供真实接口，必须拍板；否则 M1 验收范围要降级。
5. embed 最小参数集与来源必须确认：`form_id`、`project_id`、`user_id`、`user_token`。
6. `reviewerId` 兼容是否继续保留，必须确认。

## 7.1 设计基线与验收口径

- M1 发起侧页面的布局、信息层级与关键交互以 `ui/三维校审/review-designer.pen` 为主验收基线。
- 若需核对任务创建后与后续审核环节的衔接关系，可辅以 `ui/三维校审/review-flow.pen` 做跨角色流程总览核对，但不替代发起侧主设计稿。
- 本阶段涉及发起页、附件区、关联文件区的视觉与交互判断时，均以发起人视角页面是否对齐 `review-designer.pen` 为准。

## 8. 可留到 M2 / M3 的事项

以下内容可以后置，不阻塞 M1 发起侧闭环：

- `landing_role` 等更细的 embed 角色落点策略
- `/submit`、`/return` 的完整权限收口与审核人待办重构
- 发起人任务追踪大盘、退回待修改大盘、统一详情页
- WebSocket 事件体标准化与实时刷新优化
- 批注/评论/确认记录与任务详情的深度绑定
- 关联文件更复杂的分类规则和来源系统聚合策略
