# 三维校审 M6 评论协同任务单

> 生成时间：2026-03-13  
> 适用阶段：`M6：评论协同`  
> 对应范围：`ReviewCommentsPanel`  
> 主设计源：`ui/三维校审/review-reviewer.pen`  
> 总览基线：`ui/三维校审/review-flow.pen`（仅用于跨角色流程与审核工作台衔接对照）  
> 配套代码：`src/components/review/ReviewCommentsPanel.vue`、`src/composables/useToolStore.ts`、`src/composables/useReviewStore.ts`、`src/api/reviewApi.ts`

## 1. M6 目标

M6 的目标是建立正式的评论协同能力，而不是继续依赖“后端失败就静默写本地”的临时逻辑。M6 必须完成以下 5 件事：

1. 三栏评论面板职责清晰。
2. 评论只绑定 annotation，不与 workflow/record 混写。
3. 后端失败降级策略有明确产品定义。
4. 本地 store 与后端真源边界收口。
5. 评论新增、编辑、删除、回复有明确协同语义。

## 2. 范围与非范围

### 2.1 M6 范围

- `src/components/review/ReviewCommentsPanel.vue`
- `src/composables/useToolStore.ts` 中评论缓存逻辑
- `src/composables/useReviewStore.ts` 中评论通知回调
- `src/api/reviewApi.ts` 中评论接口

### 2.2 不属于 M6 的内容

## 3. 前端任务单

### FE-T6-01 三栏评论面板职责收口

- **目标**：明确 `ReviewCommentsPanel` 只负责评论展示、输入、回复、编辑、删除。
- **涉及文件**：`src/components/review/ReviewCommentsPanel.vue`
- **依赖**：角色定义、批注选中态。
- **完成标准**：
  - 输入必须依赖 `annotationType + annotationId`。
  - 无选中批注时有明确空态。
  - 三栏仅按角色展示评论。
- **风险/备注**：管理员是否归入审核栏需要拍板。

### FE-T6-02 annotation 绑定与数据装载策略收口

- **目标**：统一评论绑定键与加载策略，避免 text/cloud/rect/obb 串评论。
- **涉及文件**：`src/components/review/ReviewCommentsPanel.vue`、`src/composables/useToolStore.ts`
- **依赖**：`reviewCommentGetByAnnotation()`、annotation 结构。
- **完成标准**：
  - 按 `annotationId + annotationType` 查询。
  - 切换批注时评论正确刷新。
  - 回复链能回到正确原评论。
- **风险/备注**：annotationId 跨类型唯一性要先确认。

### FE-T6-03 后端失败降级与状态表达

- **目标**：将当前“失败即静默写本地”改为可解释的降级体验。
- **涉及文件**：`src/components/review/ReviewCommentsPanel.vue`
- **依赖**：评论接口错误码、错误提示规范。
- **完成标准**：
  - 用户能区分“已同步 / 未同步 / 同步失败”。
  - 新增、编辑、删除失败的处理口径一致。
- **风险/备注**：若仍允许本地暂存，刷新后保留/丢失规则必须明确。

### FE-T6-04 本地 store 与后端边界治理

- **目标**：将 `useToolStore` 中评论从业务真源收敛为 UI cache 或临时缓冲。
- **涉及文件**：`src/composables/useToolStore.ts`
- **依赖**：评论正式真源口径确认。
- **完成标准**：
  - `setAnnotationComments` 成为后端回填入口。
  - 本地生成 comment id 的场景只用于临时态。
  - 刷新后的行为有一致规则。
- **风险/备注**：若继续持久化评论到 localStorage，协同真假源会长期混乱。

## 4. 后端任务单

### BE-T6-01 评论模型与接口契约冻结

- **目标**：冻结评论字段与接口语义。
- **依赖**：评论 CRUD 接口现状。
- **输出**：`annotationId/annotationType/authorId/authorRole/content/replyToId/id/createdAt/updatedAt` 的必填与返回规则。
- **待确认**：是否需要补 `taskId/formId` 作为冗余索引。

### BE-T6-02 错误码与幂等语义补齐

- **目标**：让前端能基于错误码做“重试/本地暂存/阻断”判断。
- **依赖**：统一错误码骨架。
- **输出**：评论新增、编辑、删除的错误码分层与幂等/重复提交策略。
- **待确认**：删除失败时是否允许本地先删后补偿。

### BE-T6-03 评论实时事件策略确认

- **目标**：明确评论实时事件是否覆盖 `added/updated/deleted` 全量动作。
- **依赖**：用户专属 WS 通道。
- **输出**：评论事件集与最小 payload。
- **待确认**：M6 是否只要求 `comment_added`，还是同步扩展编辑/删除事件。

## 5. 联调任务单

### INT-T6-01 评论与批注联动联调

- **目标**：验证四类批注上的评论绑定、回显、切换与回复链正确性。
- **输出**：四类批注都能独立查询/新增/编辑/删除评论；切换批注不串栏、不串评论。
- **必须确认**：annotationId 稳定性与类型区分策略。

### INT-T6-02 本地/后端同步边界联调

- **目标**：验证在线、后端报错、断网、刷新页面四类场景下评论状态一致性。
- **输出**：成功同步、本地暂存、刷新后行为的联调结论。
- **必须确认**：产品是否接受离线评论持久化。

## 6. 验证任务单

### QA-T6-01 评论协同验收矩阵

- **覆盖**：无选中批注、四类批注加载、三角色发言、回复、编辑、删除、创建失败、更新失败、删除失败、刷新回显、角色限制、WS 通知。
- **通过标准**：三栏评论可用，且边界清晰、不误导用户；评论面板的视觉分栏、状态表达与交互按 `review-reviewer.pen` 验收，`review-flow.pen` 仅用于校对与审核工作台、跨角色流转的衔接关系。

## 7. M6 必须拍板的确认项

1. 评论正式主键归属是否为 `annotationId + annotationType`。
2. 后端失败降级是否允许本地暂存。
3. `useToolStore` 是否继续持久化评论。
4. 三栏角色映射是否固定为设计/校核/审核。
5. 评论实时事件是否扩展到编辑/删除。
