# 批注流程审核修复计划

> 日期：2026-04-21
> 来源：基于批注处理流程代码审核发现的 5 个问题
> 关联文档：`批注体系重构开发计划-2026-04-18.md`

## 问题清单

| # | 问题 | 严重度 | 涉及文件 |
|---|---|---|---|
| P1 | 评论 API 失败降级本地时，本地生成的 `id` 与后端不一致，后续 `loadCommentsForListView` 拉取后端数据可能导致评论重复显示 | 高 | `AnnotationPanel.vue` |
| P2 | `saveEditComment` 后端失败时本地更新无条件执行，后端拒绝（如权限不足）后前后端不一致 | 中 | `AnnotationPanel.vue` |
| P3 | `AnnotationPanel.vue` 顶部 `@ts-nocheck` 跳过类型检查，掩盖潜在类型错误 | 中 | `AnnotationPanel.vue` |
| P4 | `ConfirmedRecordData` 中 `annotations` 等字段为 `unknown[]`，丢失类型信息 | 低 | `reviewApi.ts` |
| P5 | `annotationKey` v1 (64-bit) 冲突概率需与实际业务规模确认 | 低 | `annotationKey.ts` |

## 修复方案

### P1: 评论降级时去重保护（高优）

**现状**：

```typescript
// addComment catch 分支 — 无后端 id
store.addCommentToAnnotation(type, id, {
  authorId: user.id, authorName: user.name,
  authorRole: user.role, content, replyToId,
});
// store 内部用 crypto.randomUUID() 生成本地 id
```

当后续切到 list 视图时 `loadCommentsForListView` 从后端拉取，后端返回的评论 id 与本地不同，导致同一条评论出现两次。

**修复**：

1. `addComment` 的 catch/非成功分支添加本地标记 `_localOnly: true`
2. `loadCommentsForListView` 成功后用后端数据整体替换（已有），但需确保 store 的 `setAnnotationComments` 能清除带 `_localOnly` 标记的旧记录
3. 或者更简单：catch 分支不写入 store，而是弹一个 toast 提示用户"评论发送失败，请重试"

**推荐方案**：方案 3（不写入 store + toast 提示），理由：
- 避免前后端 id 不一致的数据分叉
- 用户看到 toast 后可手动重试，UX 更清晰
- 改动最小

### P2: saveEditComment 后端失败回滚（中优）

**现状**：后端 `reviewCommentUpdate` 失败时 catch 分支只有注释，本地 `store.updateAnnotationComment` 无条件执行。

**修复**：改为乐观更新 + 失败回滚模式（与 `handleChangeSeverity` 一致）：

```typescript
async function saveEditComment() {
  // ...
  const prevContent = activeComments.value.find(c => c.id === editingCommentId.value)?.content;
  
  // 先本地更新
  store.updateAnnotationComment(type, annId, commentId, { content });
  
  try {
    const resp = await reviewCommentUpdate(commentId, content);
    if (resp && resp.success === false) {
      // 后端拒绝 → 回滚
      store.updateAnnotationComment(type, annId, commentId, { content: prevContent });
    }
  } catch {
    // 网络异常保留本地（与严重度策略一致）
  }
}
```

### P3: 移除 @ts-nocheck（中优，可渐进）

**策略**：

1. 移除 `@ts-nocheck`，先统计 TS 报错数量
2. 按报错分类处理：
   - 类型推断问题 → 补充类型标注
   - `activeAny.value` 联合类型问题 → 用类型守卫
   - props 类型问题 → 完善 `ToolsApi` 定义
3. 如果报错过多（>20处），可用 `@ts-expect-error` 逐行压制，后续逐步修

### P4: 强化 ConfirmedRecordData 类型（低优）

将 `unknown[]` 替换为具体类型：

```typescript
export type ConfirmedRecordData = {
  id?: string;
  taskId: string;
  formId?: string;
  type: 'batch';
  annotations: AnnotationRecord[];
  cloudAnnotations: CloudAnnotationRecord[];
  rectAnnotations: RectAnnotationRecord[];
  obbAnnotations?: ObbAnnotationRecord[];
  measurements: MeasurementRecord[];
  note: string;
};
```

需要确保序列化/反序列化时的字段兼容性。

### P5: annotationKey 冲突风险评估（低优，仅文档）

64-bit key 在 10k 批注下的生日悖论冲突概率 ≈ 2.7×10⁻¹²，安全。但 100k 级别上升到 ≈ 2.7×10⁻¹⁰。

**结论**：当前规模安全，无需改动。建议在文档中注明"如果批注规模预期超过 10 万条，需升级到 128-bit key 或由后端 UUID v7 接管"。
