# 三维校审 M6 评论协同会议 Checklist

> 用途：开会快速对齐 + 会后直接派单  
> 主设计源：`ui/三维校审/review-reviewer.pen`  
> 总览基线：`ui/三维校审/review-flow.pen`（仅用于跨角色流程与审核工作台衔接对照）  
> 对应文档：`开发文档/三维校审/三维校审M6评论协同任务单.md`

## 一、会议目标

- [ ] 对齐 M6 只解决“评论协同正式化”，不扩到 M7
- [ ] 冻结评论绑定、失败降级、本地 store 边界、实时事件范围
- [ ] 明确三栏评论的正式角色映射与协同语义
- [ ] 会后可以直接按 owner 派单开工

## 二、会前必读

- [ ] 阅读 `开发文档/三维校审/三维校审M6评论协同任务单.md`
- [ ] 阅读 `开发文档/三维校审/三维校审重构执行方案.md`
- [ ] 对照 `ui/三维校审/review-reviewer.pen` 确认评论面板主设计与角色分栏
- [ ] 如需核对与审核工作台、跨角色流程的衔接，再参考 `ui/三维校审/review-flow.pen`
- [ ] 前端确认范围落点：`ReviewCommentsPanel` / `useToolStore` / `useReviewStore`
- [ ] 后端准备评论 CRUD、错误码、评论实时事件现状

## 三、必须拍板项

- [ ] 评论正式绑定键是否统一为 `annotationId + annotationType`
- [ ] 后端失败时是否允许本地暂存；若允许，刷新后是否保留
- [ ] `useToolStore` 中评论是否只保留为 UI cache，不再作为真源
- [ ] 三栏角色映射是否固定为设计 / 校核 / 审核
- [ ] 实时事件是否仅做新增，还是覆盖新增 / 编辑 / 删除

## 四、前端 Owner 清单

- [ ] FE1：收口 `ReviewCommentsPanel` 职责，只做展示、输入、回复、编辑、删除
- [ ] FE2：切换 text/cloud/rect/obb 时评论正确刷新，不串评论、不串回复链
- [ ] FE3：新增、编辑、删除失败时统一展示“已同步 / 未同步 / 同步失败”
- [ ] FE4：收敛本地评论缓存语义，避免本地 id/持久化继续污染真源

## 五、后端 Owner 清单

- [ ] BE1：冻结评论模型与字段契约
- [ ] BE2：补齐评论新增、编辑、删除的错误码与幂等策略
- [ ] BE3：确认评论实时事件最小范围与 payload
- [ ] BE4：明确是否补 `taskId/formId` 作为评论冗余索引

## 六、联调 / 测试清单

- [ ] INT1：四类批注分别验证评论加载、新增、回复、编辑、删除
- [ ] INT2：验证批注切换后评论正确刷新，回复链能回到原评论
- [ ] INT3：覆盖在线、后端报错、断网、刷新页面四类场景
- [ ] QA1：验证三角色发言展示、权限限制、通知回流是否一致
- [ ] QA1-视觉基线：评论面板分栏、状态表达与交互按 `review-reviewer.pen` 验收
- [ ] QA2：验证成功同步、本地暂存、刷新后行为是否符合拍板结论

## 七、风险提醒

- [ ] 风险：annotationId 跨类型不唯一会直接串评论
- [ ] 风险：继续本地持久化评论会长期制造真假源混乱
- [ ] 风险：失败降级口径不统一，用户会误判“已提交成功”
- [ ] 风险：实时事件范围不明确，后续联调会反复返工
