## 二、审核发现

### CRITICAL — 需要尽快修复

#### C-1: `checker_name` 和 `approver_name` 绑定了 ID 而非姓名

**文件**: `plant-model-gen/src/web_api/review_api.rs` (L806-808)

```rust
.bind(("checker_name", checker_id.clone()))    // ← 应该绑定 checker 姓名
.bind(("approver_id", approver_id.clone()))
.bind(("approver_name", approver_id.clone()))   // ← 应该绑定 approver 姓名
```

`create_task` 函数在创建任务时，将 `checker_name` 和 `approver_name` 都绑定为对应的 ID 字符串。这导致前端 `ReviewerTaskList` 和 `ReviewPanel` 中显示审核人员时会展示 ID 而非真实姓名。

**建议**: 前端 `CreateTaskRequest` 应增加 `checkerName` / `approverName` 字段，后端从请求中读取并存储；或者后端根据 ID 查询用户表获取姓名。

---

#### C-2: `requester_name` 直接使用 `user_id`

**文件**: `plant-model-gen/src/web_api/review_api.rs` (L752)

```rust
let requester_name = claims.user_id.clone();
```

创建任务时 `requester_name` 始终等于 `user_id`，在 UI 上会显示 ID 而非用户昵称。

**建议**: JWT claims 中增加 `user_name` 字段，或在创建任务时查询用户信息。

---

#### C-3: `embed-url` 接口关闭了 token 校验

**文件**: `plant-model-gen/src/web_api/model_center_client.rs` (L271-280)

```rust
// TODO: 暂时关闭 token 验证，后续需恢复
let mut jwt_claim_form_id: Option<String> = None;
if let Some(ref token) = request.token {
    let token = token.trim();
    if token.split('.').count() == 3 {
        if let Ok(claims) = decode_token_unsafe(token) {  // ← 不验签
            jwt_claim_form_id = Some(claims.form_id);
        }
    }
}
```

`get_embed_url` 使用 `decode_token_unsafe`（关闭签名验证和过期检查）。任何人都可以伪造一个 JWT 传入任意 `form_id`。

**建议**: 上线前必须恢复签名校验（`decode_token` 而非 `decode_token_unsafe`），同时在 `embed-url` 路由上应用认证中间件或自行校验请求级 token。

---

#### C-4: `embed-url` 和 `workflow/sync` 路由不在 JWT 中间件保护范围内

**文件**: `plant-model-gen/src/web_api/model_center_client.rs` 的 `create_model_center_routes()`

`model_center_routes` 单独注册，没有挂载 `review_auth_middleware`。对比 `review_api_routes` 则挂载了 JWT 中间件。这意味着 `/api/review/embed-url`、`/api/review/workflow/sync`、`/api/review/delete`、`/api/review/cache/preload` 完全不受 JWT 保护。

当前这些接口使用 SHA256 token 做请求级鉴权，但 `embed-url` 的 SHA256 校验已被注释掉（见 C-3）。

**建议**: 明确文档化哪些接口走 JWT、哪些走 SHA256 请求签名，确保每条路由至少有一种有效鉴权。

#### W-4: 异步通知失败无重试机制

**文件**: `plant-model-gen/src/web_api/model_center_client.rs` (L1175-1202)

```rust
tokio::spawn(async move {
    let result = notify_workflow_sync(&task_id, &action, &operator_id, comment.as_deref()).await;
    match result {
        Ok(_) => info!(...),
        Err(e) => warn!(...),  // 仅日志，无重试
    }
});
```

`notify_workflow_sync_async` 是 fire-and-forget 模式。如果外部系统暂时不可达或返回错误，通知会静默丢失。

**建议**: 
- 短期：增加有限次数的重试（如 3 次指数退避）
- 中期：引入消息队列或持久化通知记录表，失败后可手动/定时重发

#### W-5: `submit_to_next_node` 权限模型不完整

**文件**: `plant-model-gen/src/web_api/review_api.rs` (L2179-2188)

```rust
let has_permission = match current_node.as_str() {
    "sj" => task_row.requester_id.as_deref() == Some(operator_user),
    "jd" => task_row.checker_id.as_deref() == Some(operator_user)
         || task_row.reviewer_id.as_deref() == Some(operator_user),
    "sh" => task_row.approver_id.as_deref() == Some(operator_user),
    "pz" => task_row.approver_id.as_deref() == Some(operator_user),  // sh 和 pz 都检查 approver_id
    _ => false,
};
```

`sh`（审核）和 `pz`（批准）节点使用同一个 `approver_id` 做权限判断。如果审核和批准是不同人员，当前模型不支持。这与文档中描述的"四节点"模式存在 gap：缺少独立的 `pz` 节点负责人字段。

**建议**: 
- 如果 `sh` 和 `pz` 确实是同一人，应在文档中说明
- 如果需要分离，需新增 `final_approver_id` 字段

#### W-6: 前端存在冗余的旧版 API（`approve`、`reject`）

**文件**: `plant3d-web/src/api/reviewApi.ts` (L329-357)

前端同时暴露了旧版 `reviewTaskApprove`/`reviewTaskReject` 和新版 `reviewTaskSubmitToNext`/`reviewTaskReturn`。后端路由同样注册了 `/approve` 和 `/reject`（L686-687）与 `/submit` 和 `/return`（L690-691）。

**建议**: 统一迁移到新版多级审批 API（`submit`/`return`），并在界面层统一为“确认提交流转 / 确认驳回流转”口径，废弃旧版单级审批接口，避免混用导致状态不一致。

### INFO — 需知晓

#### I-1: 前端 API 层的数据规范化很完善

`reviewApi.ts` 的 `normalizeReviewTask` 函数处理了 `snake_case` / `camelCase` 双重命名、时间戳格式兼容（秒/毫秒/ISO字符串）、`reviewer_id` 到 `checker_id` 的旧字段映射。代码质量良好。

#### I-2: 路由参数编码正确

所有 `fetchJson` 调用中的路径参数都使用了 `encodeURIComponent`，防止了 ID 中特殊字符导致的路由问题。

#### I-3: JWT 中间件支持调试模式

`review_auth_middleware` 在 `enabled=false` 时仍会尝试解析 Bearer token（使用 `decode_token_unsafe`），解析失败才回退到 debug claims。这个设计对本地开发友好。

#### I-4: WebSocket URL 构建逻辑合理

`getReviewWebSocketUrl` 和 `getReviewUserWebSocketUrl` 支持多种环境变量覆盖，HTTP/HTTPS 到 WS/WSS 的协议转换处理正确。

#### I-5: 附件上传支持进度回调

`reviewAttachmentUploadWithProgress` 使用 XHR 实现了上传进度监控，对大文件上传的用户体验有帮助。

#### I-6: `review_form_model` 去重处理

创建任务时使用 `HashSet` 对构件 `refNo` 去重后再写入 `review_form_model`，避免了重复关联记录。

#### I-7: 任务列表查询有容错恢复

`list_tasks` 在分页查询返回空但 total > 0 时会降级为逐行探测（`recover_review_task_page_with_row_probes`），这对 SurrealDB 可能的反序列化问题提供了有效兜底。
