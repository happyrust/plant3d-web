> 生成时间：2026-03-13  
> 适用范围：`plant3d-web` 三维校审前端工作台、任务流转、批注协同、附件与辅助校审数据闭环  
> 设计输入：`walkthrough.md.resolved`、`implementation_plan.md.resolved`、`ui/三维校审/review-designer.pen`、`ui/三维校审/review-reviewer.pen`、`ui/三维校审/review-flow.pen`

## 1. 背景与目标

当前三维校审能力已经具备任务创建、任务列表、流转、批注评论、确认记录、附件上传、辅助校审数据和嵌入式访问等基础能力，但实现仍存在以下结构性问题：

- 任务模型中 `checkerId` / `approverId` 与兼容字段 `reviewerId` 并存，语义尚未完全收口，见 `src/types/auth.ts`、`src/api/reviewApi.ts`。
- 前端同时维护本地模式与后端模式，`useUserStore`、`useReviewStore` 中存在较多 `localStorage` 与 `USE_BACKEND` 分支，导致状态来源分裂，见 `src/composables/useUserStore.ts`、`src/composables/useReviewStore.ts`。
- 流转接口同时存在 `/submit`、`/return` 与旧 `/approve`、`/reject`、`/start-review`，页面行为与后端契约容易出现重复表达，见 `src/api/reviewApi.ts`。
- 设计稿已经定义了完整的 Phase 1-5 界面，但当前实现仍包含旧式工作台、调试型辅助数据面板和未完成拆分的批注交互。

本次重构目标如下：

1. 建立统一的三维校审业务闭环：发起编校审 -> 校对 -> 审核 -> 批准 -> 退回重提。
2. 以设计稿为准重构核心界面，并清理旧逻辑残留。
3. 收敛任务、流转、批注、评论、确认记录、附件、辅助数据与通知的数据边界。
4. 为后续并行开发提供可执行的阶段计划、职责拆分和验收口径。

## 3. 现状问题总结

### 3.1 前端结构问题

- `ReviewPanel` 承担了任务摘要、流转、确认记录、碰撞查询、辅助数据与局部交互，职责过重，见 `src/components/review/ReviewPanel.vue`。
- `AnnotationPanel` 仍承担旧式批注工具、批注列表和评论操作，不符合设计稿中“悬浮工具栏 + 独立列表面板”的职责拆分，见 `src/components/tools/AnnotationPanel.vue`。
- `ReviewAuxData` 仍暴露 `project_id`、`form_id`、`UCode`、`UKey` 等调试参数，不是正式产品化 UI，见 `src/components/review/ReviewAuxData.vue`。
- `TaskReviewDetail`、`DesignerTaskList`、`ResubmissionTaskList` 已有雏形，但信息组织仍偏旧式 CRUD 与过滤列表，不符合新的工作台化设计。

### 3.2 后端契约与状态机问题

- `ReviewTask` 已同时包含 `checkerId`、`approverId`、`currentNode` 与兼容字段 `reviewerId`，前端创建与读取时仍在双写/双读，见 `src/types/auth.ts`、`src/composables/useUserStore.ts`。
- `status` 与 `currentNode` 存在双真源风险。当前前端既按 `status` 过滤列表，也按 `currentNode` 决定按钮文案和流转方向，容易出现显示正确但权限错误的问题。
- 旧接口 `/start-review`、`/approve`、`/reject` 与新接口 `/submit`、`/return` 同时存在，业务表达重叠，见 `src/api/reviewApi.ts`。

### 3.3 数据边界问题

- 评论数据挂在批注对象下，但当前 `ReviewCommentsPanel` 在后端失败时会降级保存到本地 store，存在协同数据不一致风险，见 `src/components/review/ReviewCommentsPanel.vue`。
- 确认记录以批量快照方式保存文字批注、云线、矩形、OBB 和测量数据，边界尚可，但需要与审批流转历史保持明确分离，见 `src/api/reviewApi.ts`、`src/composables/useReviewStore.ts`。
- embed 流程依赖 `form_id`、`user_token`、`project_id` 等参数，同时部分行为借助 `sessionStorage` 辅助落点，仍需进一步标准化，见 `src/api/reviewApi.ts`、`src/components/review/embedRoleLanding.ts`。

## 4. 目标业务闭环与核心原则

### 4.1 目标业务闭环

目标业务链路为：

1. 设计/编制人员在 `InitiateReviewPanel` 中选择构件、填写数据包、上传附件并创建编校审单。
2. 任务进入工作流，由 `ReviewerTaskList` 中对应节点处理人领取并进入 `ReviewPanel`。
3. 审核人员在工作台中执行批注、评论、测量、确认记录、查看辅助校审数据，并决定确认流转或退回。
4. 发起人在 `DesignerTaskList` / `ResubmissionTaskList` 中查看状态、退回原因和历史记录，修改后再次提交。
5. 整个过程中，`TaskReviewDetail`、`ReviewCommentsPanel`、`AssociatedFilesList` 共同承担追踪、解释和留痕职责。

### 4.2 核心原则

- **单一任务真源**：`taskId` 作为技术主键，`formId` 作为业务单号，二者并存但职责清晰。
- **节点驱动流程**：`currentNode` 是工作流真源，`status` 只表达生命周期与展示态。
- **评论、确认、流转分治**：批注评论不混入流转历史，确认记录不充当审批动作。
- **设计优先，契约先行**：先统一接口与字段，再落地新界面，避免“先换皮后返工”。
- **兼容迁移，不做一次性切断**：`reviewerId`、旧接口与旧鉴权模式进入兼容期，但需明确废弃计划。

## 5. 前后端字段与接口迁移策略

### 5.1 字段迁移策略

| 类别 | 当前字段 | 目标字段/语义 | 迁移策略 |
| --- | --- | --- | --- |
| 任务主键 | `id` | 技术主键 | 保留 |
| 业务单号 | `formId` / `form_id` | 编校审单/表单业务主键 | 保留，接口双读双写兼容 |
| 校对人员 | `checkerId` / `checkerName` | 第二节点处理人 | 保留，作为正式字段 |
| 审核人 | `approverId` / `approverName` | 第三节点处理人 | 保留，作为正式字段 |
| 兼容审核人 | `reviewerId` / `reviewerName` | 历史 reviewer 语义 | 兼容期双读，创建时允许 `reviewerId=checkerId`，最终删除 |
| 当前节点 | `currentNode` | `sj/jd/sh/pz` | 保留，作为工作流真源 |
| 生命周期状态 | `status` | `draft/submitted/in_review/approved/rejected/cancelled` | 保留，但展示文案可重映射 |
| 退回原因 | `returnReason` | 退回说明 | 保留，强化前端展示 |
| 流转历史 | `workflowHistory` | 只记录节点动作 | 保留，不混入评论与确认 |

### 5.2 接口迁移策略

#### 正式保留接口

- `POST /api/review/tasks`
- `GET /api/review/tasks`
- `GET /api/review/tasks/{taskId}`
- `PATCH /api/review/tasks/{taskId}`
- `POST /api/review/tasks/{taskId}/submit`
- `POST /api/review/tasks/{taskId}/return`
- `GET /api/review/tasks/{taskId}/workflow`
- `POST /api/review/comments`
- `POST /api/review/records`
- `POST /api/review/attachments`
- `POST /api/review/embed-url`
- `POST /api/review/cache/preload`

#### 兼容期保留接口

- `POST /api/review/tasks/{taskId}/start-review`
- `POST /api/review/tasks/{taskId}/approve`
- `POST /api/review/tasks/{taskId}/reject`

兼容期内约定如下：

- 新页面与新工作流默认优先调用 `/submit`、`/return`。
- `/approve` 仅保留给最终批准节点或历史兼容入口，具体是否保留为独立动作，待后端确认。
- `/reject` 视作 `/return targetNode=sj` 的兼容别名，后续下线。
- `/start-review` 若后端可自动在进入工作台时标记 `in_review`，则后续下线。

### 5.3 WebSocket 与通知迁移

- 当前 `useReviewStore` 使用 `task.reviewerId` 建立用户 WS 通道，后续应改为当前登录用户 `userId`，见 `src/composables/useReviewStore.ts`。
- 事件体建议统一补齐：`taskId`、`formId`、`currentNode`、`status`、`operatorId`。
- 建议新增或标准化以下事件：`task_node_changed`、`task_returned`、`record_created`、`comment_added`。

### 5.4 embed 与辅助数据迁移

- embed URL 继续保留 `form_id`、`user_token`、`user_id`、`project_id`、`output_project`。
- 建议新增标准参数 `landing_role` 或等效字段，以区分发起人、校对人员、审核人、批准人落点，待后端确认。
- `/api/review/aux-data` 当前仍依赖 `UCode` / `UKey` Header。迁移目标为统一业务 token 鉴权；兼容期可双模式支持。
- `form_id` 必须优先使用 `task.formId`，不得默认回落 `task.id` 作为正式业务逻辑。

## 9. 风险、依赖与验收标准

### 9.1 主要风险

- `reviewerId` 兼容字段残留过深，可能导致角色分流、WS 建链和任务过滤错误。
- `status` 与 `currentNode` 并存期间，页面筛选和按钮权限判断可能出现不一致。
- `/approve`、`/reject`、`/start-review` 与 `/submit`、`/return` 的语义重叠，若不先澄清，联调阶段会频繁返工。
- `ReviewAuxData` 依赖的鉴权模式和参数来源仍偏临时，正式化前需要后端确认。
- `ReviewCommentsPanel` 与批注实体的持久化边界仍需后端确认，特别是是否允许本地降级保存。

### 9.2 关键依赖

- 后端确认正式字段、兼容字段和废弃字段的迁移窗口。
- 后端确认 embed 返回体、辅助数据鉴权与工作流事件体。
- viewer 侧确认批注工具、定位、高亮、确认记录回写的交互接口。

### 9.3 验收标准

#### 功能验收

- 能完整跑通 `sj -> jd -> sh -> pz` 四段流程。
- 退回重提链路可追踪、可解释、可再次提交流转。
- 批注、评论、确认记录、附件与辅助数据均能在任务上下文中被正确查看和回放。

#### 数据验收

- `taskId`、`formId`、`currentNode`、`workflowHistory` 在列表、详情、工作台中保持一致。
- 评论、确认记录、流转历史三条数据线不混写、不丢失。
- 嵌入参数与任务上下文一致，不依赖隐式本地状态才能打开正确面板。

#### 视觉与交互验收

- 发起人页面与 `review-designer.pen` 保持一致的层级、布局与关键交互。
- 审核人页面与 `review-reviewer.pen` 保持一致的层级、布局与关键交互。
- 跨角色流程关系以 `review-flow.pen` 为总览基线。
- 批注工具栏、批注列表、流转弹窗、退回大盘和辅助数据面板不再使用旧式临时交互。
