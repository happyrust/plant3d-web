# 批注体系重构开发计划

> 日期：2026-04-18  
> 适用范围：`plant3d-web` 三维校审批注、评论、确认记录、回放恢复、工作流同步与嵌入恢复链路  
> 关联文档：`新的三维校审流程分析.md`、`校审流程分析.md`、`三维校审重构执行方案.md`、`编校审交互接口设计.md`

## 1. 背景

当前三维校审中的批注体系已经具备以下基础能力：

- 在 `ReviewPanel` 内直接发起文字批注、云线批注、矩形批注与测量。
- 通过 `useToolStore` 维护前端工具态与本地持久化。
- 通过 `ReviewCommentsTimeline` 和 `/api/review/comments*` 实现批注级评论协作。
- 通过 `/api/review/records*` 将批注与测量以 `ConfirmedRecord` 批量快照形式持久化。
- 通过 `confirmedRecordsRestore.ts` 与 `reviewRecordReplay.ts` 恢复确认记录到场景。
- 通过 `/api/review/workflow/sync` 在嵌入/PMS 场景下按 `form_id` 恢复 `records + annotationComments + attachments` 聚合快照。

这套实现已经能支撑当前校审主链路，但随着协作深度、跨系统同步与后续正式产品化需求增加，现有实现暴露出明显的结构性问题：

1. **批注、评论、确认记录、workflow/sync 快照分层不彻底**，部分数据边界靠组件约定维持。
2. **平台内恢复链与外部嵌入恢复链是两套语义**，容易出现同任务不同入口视图不一致。
3. **工具草稿态与已确认态未分层**，当前依赖 `clearAll() + replay` 维持场景一致性，扩展性有限。
4. **批注评论依附于 annotation 对象内部数组**，短期方便，长期不利于协作扩展、分页、增量同步与已读态。
5. **缺少稳定的批注业务锚点**，评论与跨批次批注演进之间存在漂移风险。

因此需要一份完整的开发计划，指导后续对批注体系进行分阶段重构。

## 2.2 当前数据层分工

### A. 工具草稿层

由 `useToolStore.ts` 维护：

- `annotations`
- `cloudAnnotations`
- `rectAnnotations`
- `obbAnnotations`
- `measurements`
- `xeokitDistanceMeasurements`
- `xeokitAngleMeasurements`

这一层主要服务于用户“正在画、正在改、还没确认”的前端交互。

### B. 批注评论层

由 `ReviewCommentsTimeline.vue` + `/api/review/comments*` 驱动，主键为：

- `annotationId`
- `annotationType`

这一层服务于围绕某条批注的协作讨论。

### C. 确认记录层

由 `useReviewStore.addConfirmedRecord()` + `/api/review/records*` 驱动，主键为：

- `taskId`

这一层服务于任务级审核证据沉淀。

### D. workflow/sync 聚合层

由 `/api/review/workflow/sync` 驱动，主键为：

- `form_id`

可返回：

- `records`
- `annotationComments`
- `attachments`
- `models`
- `currentNode`
- `taskStatus`

这一层服务于嵌入/PMS/跨系统恢复与同步。

## 2.3 当前调用链

### 平台内工作台

1. `ReviewerTaskList` 选择任务。
2. `reviewStore.setCurrentTask(task)`。
3. 自动调用：
   - `loadConfirmedRecords(task.id)`
   - `loadReviewHistory(task.id)`
4. `confirmedRecordsRestore` 监听任务与记录变化。
5. 使用 `buildReviewRecordReplayPayload(records)` 将确认记录灌回 `toolStore`。

### 工作台内评论

1. `ReviewPanel` 展开批注详情。
2. `ReviewCommentsTimeline` 按 `annotationId + annotationType` 拉评论。
3. 调用：
   - `reviewCommentGetByAnnotation`
   - `reviewCommentCreate`
   - `reviewCommentUpdate`
   - `reviewCommentDelete`
4. 结果再写回 `toolStore` 中对应 annotation 的 `comments`。

### 工作台内确认

1. `ReviewPanel.confirmCurrentData()`
2. `buildReviewConfirmSnapshotPayload(...)`
3. `reviewStore.addConfirmedRecord(...)`
4. `reviewRecordCreate('/api/review/records')`
5. 成功后 `toolStore.clearAll()`
6. 再通过 `restoreConfirmedRecordsIntoScene(true)` 重建场景

### 嵌入恢复

1. `restoreEmbedFormSnapshot()`
2. `reviewWorkflowSyncQuery({ formId, token, actor })`
3. 得到 `records + annotationComments`
4. `buildWorkflowSnapshotReplayPayload(records, comments)`
5. `toolStore.importJSON(payload)`
6. `viewerTools.syncFromStore()`

## 3. 问题与缺口

## 3.1 数据边界问题

### 3.1.1 评论与确认记录之间缺少稳定关联锚点

当前评论依赖 `annotationId + annotationType`，而批注本身可能随着多轮确认、导入回放、跨系统恢复发生重建、覆盖或换 id。

风险：

- 评论归属可能漂移。
- 同一业务批注跨确认批次后无法稳定追踪。
- 后续做“本轮意见”“上一轮意见”时难以归并。

### 3.1.2 annotation.comments 结构耦合过深

评论直接塞进 annotation record：

- 短期降低了实现成本。
- 长期会导致 annotation 结构越来越重。
- replay / import / export 都要额外承担评论 merge 逻辑。

### 3.1.3 draft 与 confirmed 未分层

确认成功后直接 `toolStore.clearAll()` 再恢复确认态，会导致：

- 模型语义上只有“一份当前工具态”。
- 后续难以支持“未确认草稿”和“已确认证据”同时显示。
- 难以扩展局部确认、分批确认、草稿箱与多人协同编辑。

## 3.2 恢复语义问题

### 3.2.1 内部恢复与外部恢复不统一

当前存在两套恢复：

- 平台内：`taskId -> confirmedRecords`
- 外部嵌入：`formId -> workflow/sync(records + comments + attachments)`

风险：

- 同一任务从不同入口进入，看到的数据可能不同。
- 后续定位“哪个视图是权威视图”会越来越困难。

### 3.2.2 replay payload 既承担渲染恢复又承担数据归一

`reviewRecordReplay.ts` 同时负责：

- records 合并
- comments 绑定
- measurements 归一
- JSON payload 生成

职责偏重，未来若再加入评论分页、附件锚点、节点级过滤，会继续膨胀。

## 3.3 协同一致性问题

### 3.3.1 comment_added WebSocket 链路不完整

目前 `useReviewStore.ts` 有 `comment_added` 事件，但缺少统一的“收到后如何刷新 annotation comments”的全局机制。

风险：

- 多人协同时评论列表更新不一致。
- 组件各自处理后容易出现重复逻辑。

### 3.3.2 本地存储作用域不够细

`useToolStore` 当前 scope 主要按 project/db 切分，而不是 task 粒度。

风险：

- 同项目多任务之间 draft 批注串扰。
- 场景切换后出现错误草稿回显。

## 3.4 领域建模问题

### 3.4.1 缺少 workflow node / review round 显式字段

当前批注与评论没有稳定表达：

- 属于哪个节点阶段
- 属于第几轮退回/重提
- 属于谁的操作回合

这会影响：

- 节点级过滤
- 分栏展示（设/校/审/批）
- 按轮次对账
- 外部系统归档

### 3.4.2 缺少事件流/增量变更层

当前只有 batch snapshot，没有 annotation event stream。

后果：

- 不能准确回答“哪一轮新增了哪条批注”
- 不能较低成本做差异对比
- 审计只能靠快照比对，成本高

## 4. 重构目标

本次批注体系重构目标分为五类：

### 4.1 统一数据分层

明确拆分：

- 批注实体层（annotation entity）
- 评论协作层（comment thread）
- 任务确认层（confirmed snapshot）
- 聚合快照层（review snapshot）

### 4.2 统一恢复模型

无论来自：

- 平台任务内部
- workflow/sync 嵌入恢复
- 后端导入导出

都统一转成同一套 `ReviewSnapshot` 结构再驱动 UI。

### 4.3 强化协同一致性

- 评论实时同步统一化
- 批注与评论主键稳定化
- 多入口恢复结果一致化

### 4.4 支撑后续产品化能力

为以下能力提前建模：

- 评论分页/已读/@人
- 批注节点级过滤
- 按轮次查看校审记录
- 局部确认 / 分批确认
- 差异比对 / 审计追踪

### 4.5 避免一次性推翻

保留兼容期：

- 当前 `/api/review/records`
- 当前 `/api/review/comments`
- 当前 replay JSON 格式

通过“新增中间层 + 渐进式迁移”完成演进。

## 5. 目标架构

## 5.1 目标分层模型

### A. Annotation Entity

表示一条批注本体，建议统一字段：

- `annotationKey`：稳定业务键
- `annotationId`：当前存储实体 id
- `taskId`
- `formId`
- `workflowNode`
- `reviewRound`
- `type`
- `geometryPayload`
- `contentPayload`
- `author`
- `createdAt`
- `updatedAt`
- `status`

说明：

- `annotationKey` 用于跨快照/跨同步归并。
- `annotationId` 保持现有后端与前端兼容。

### B. Annotation Comment Thread

单独建模评论线程：

- `commentId`
- `annotationKey`
- `annotationId`
- `taskId`
- `formId`
- `workflowNode`
- `reviewRound`
- `author`
- `content`
- `replyToId`
- `createdAt`
- `updatedAt`

说明：

- UI 不再依赖 `annotation.comments` 作为唯一真源。
- `annotation.comments` 在兼容期只作为渲染投影，不再作为存储真源。

### C. Confirmed Snapshot

用于表达某次任务内确认动作：

- `recordId`
- `taskId`
- `formId`
- `workflowNode`
- `reviewRound`
- `annotations[]`
- `measurements[]`
- `note`
- `confirmedAt`
- `confirmedBy`

### D. Review Snapshot

统一恢复/渲染的中间结构：

- `source`: `task_records | workflow_sync | import_package`
- `taskId`
- `formId`
- `workflowNode`
- `taskStatus`
- `annotations`
- `comments`
- `measurements`
- `attachments`
- `models`
- `meta`

所有外部数据源都先转成 `ReviewSnapshot`，UI 与 viewer 只消费这一层。

## 5.2 目标前端分层

建议新增以下模块：

- `src/review/domain/annotationEntity.ts`
- `src/review/domain/commentThread.ts`
- `src/review/domain/reviewSnapshot.ts`
- `src/review/adapters/reviewRecordAdapter.ts`
- `src/review/adapters/workflowSyncAdapter.ts`
- `src/review/adapters/toolStoreAdapter.ts`
- `src/review/services/reviewSnapshotService.ts`
- `src/review/services/commentSyncService.ts`

说明：

- `adapters` 负责把现有 API/本地结构适配成新领域模型。
- `services` 负责流程编排与统一恢复。

## 5.3 目标 viewer 恢复链

目标链路：

1. 读取任意源数据
2. 转为 `ReviewSnapshot`
3. 由 `toolStoreAdapter` 转为 viewer 可消费的图元 payload
4. 分层渲染：
   - confirmed layer
   - draft layer
   - comment overlay / selected annotation overlay

## 6. 分阶段开发计划

## 6.1 Phase A：基线梳理与冻结

目标：把当前链路冻结成可迁移状态。

### 开发项

1. 盘点当前批注、评论、确认记录、workflow/sync 字段映射。
2. 确认 `annotationId` 在平台内/外部恢复中的稳定性现状。
3. 输出统一术语表：
   - annotationId
   - annotationKey
   - taskId
   - formId
   - workflowNode
   - reviewRound
4. 评估后端是否可在评论表/记录表补充新字段。

### 输出物

- 字段映射表
- 兼容性清单
- 风险矩阵

### 验收标准

- 所有现有链路的数据来源、消费点、主键关系有清单可查。
- 前后端对新增字段达成统一口径。

## 6.2 Phase B：引入统一 ReviewSnapshot 中间层

目标：让平台内恢复与外部恢复统一走一个中间结构。

### 开发项

1. 新增 `ReviewSnapshot` 类型定义。
2. 新增：
   - `reviewRecordAdapter`
   - `workflowSyncAdapter`
3. 把以下链路改为先产出 `ReviewSnapshot`：
   - `confirmedRecordsRestore.ts`
   - `embedFormSnapshotRestore.ts`
   - `reviewRecordReplay.ts`
4. 保持现有 viewer 恢复结果不变。

### 重点修改文件

- `src/components/review/reviewRecordReplay.ts`
- `src/components/review/confirmedRecordsRestore.ts`
- `src/components/review/embedFormSnapshotRestore.ts`
- 新增 `src/review/domain/*`
- 新增 `src/review/adapters/*`

### 验收标准

- task 内恢复与 workflow/sync 恢复走同一套 snapshot 转换。
- 原有测试与现有 UI 不退化。

## 6.3 Phase C：评论从 annotation 内联结构解耦

目标：把评论从 annotation record 内剥离为独立真源。

### 开发项

1. 新增 comment thread store/index：
   - 按 `annotationKey` / `annotationId` 建索引
2. `ReviewCommentsTimeline` 改为直接读 comment store
3. `useToolStore` 中 `comments` 字段保留兼容投影，不再作为真源
4. `mergeWorkflowCommentsIntoRecords()` 改为先归入 comment store，再投影到 UI
5. 为后续分页/已读保留扩展位

### 重点修改文件

- `src/composables/useToolStore.ts`
- `src/components/review/ReviewCommentsTimeline.vue`
- `src/components/review/reviewRecordReplay.ts`
- 新增 `src/review/services/commentSyncService.ts`

### 验收标准

- 评论读写不再依赖 annotation.comments 作为唯一真源。
- 回放恢复后评论仍能正确展示。

## 6.4 Phase D：引入 annotationKey 与轮次字段

目标：让评论、批注、确认记录拥有稳定业务锚点。

### 开发项

1. 为 annotation 引入稳定 key 生成策略：
   - 初始兼容期可在前端生成
   - 长期建议后端生成并回写
2. 评论请求体与响应体补充：
   - `annotationKey`
   - `taskId`
   - `formId`
   - `workflowNode`
   - `reviewRound`
3. 确认记录中 annotation payload 也补充同样字段
4. workflow/sync query 返回的 comments/records 尝试带出新字段

### 后端配合需求

- 评论表补新字段
- 记录表支持透传 annotationKey/workflowNode/reviewRound
- workflow/sync 聚合返回统一字段

### 验收标准

- 评论与批注可稳定跨快照归并。
- 同一任务多轮退回/重提后仍可区分轮次。

## 6.5 Phase E：draft/confirmed 双层渲染

目标：替换当前 `clearAll() + replay` 的硬切模式。

### 开发项

1. 在前端 viewer 层分离：
   - `draft annotation layer`
   - `confirmed annotation layer`
2. `confirmCurrentData()` 改为：
   - 从 draft layer 生成确认记录
   - 成功后仅清空 draft layer
   - 不再全量清空所有工具数据
3. 调整 `toolStore` 结构，使其能表达双层状态
4. 兼容当前 UI 计数逻辑：
   - pending count 只统计 draft
   - confirmed list 只统计 confirmed records

### 重点修改文件

- `src/composables/useToolStore.ts`
- `src/components/review/ReviewPanel.vue`
- viewer / scene sync 相关 composable

### 验收标准

- 确认后场景仍正确显示已确认数据。
- 新增草稿不会污染已确认层。
- 支持同任务下继续画新批注而不影响历史确认层。

## 6.6 Phase F：评论实时同步统一化

目标：统一 comment_added 的实时刷新策略。

### 开发项

1. 建立统一 `commentSyncService`
2. WebSocket 收到 `comment_added` 后：
   - 优先增量更新 comment store
   - 必要时回拉对应 annotation 的评论线程
3. 避免组件级重复实现
4. 对评论编辑/删除也补充实时事件预留

### 重点修改文件

- `src/composables/useReviewStore.ts`
- `src/components/review/ReviewCommentsTimeline.vue`
- 新增 `src/review/services/commentSyncService.ts`

### 验收标准

- A 用户新增评论，B 用户无刷新即可看到。
- 不出现重复评论、乱序评论。

## 6.7 Phase G：task 级草稿作用域隔离

目标：避免同项目多任务之间的批注草稿串扰。

### 开发项

1. 调整 `useToolStore` 的本地持久化 scope：
   - 从 `project + db`
   - 升级为 `project + db + taskId`（校审模式下）
2. 非校审场景保留原有 scope
3. 任务切换时明确：
   - 是否加载当前任务 draft
   - 是否保留通用建模工具数据

### 验收标准

- 不同任务切换后，不会错误继承他任务草稿。
- 当前任务草稿仍能恢复。

## 6.8 Phase H：事件流与差异审计增强（可选增强）

目标：补充增量审计层，为后续对比分析与高级追踪提供基础。

### 开发项

1. 引入 annotation event model：
   - added
   - updated
   - deleted
   - commented
   - confirmed
2. 评估是否新增后端事件表
3. 提供“本轮新增/修改/删除”差异视图基础能力

### 说明

此阶段不是当前主链路上线前置，可在前述阶段稳定后开展。

## 8. 风险评估

## 8.1 高风险项

### A. 批注稳定 key 的生成策略

若前后端对 `annotationKey` 口径不统一，会造成评论仍旧无法可靠归并。

应对：

- 先定义明确生成规则
- 兼容期允许双字段并存
- 优先以后端回写为准

### B. draft/confirmed 双层改造影响 viewer 同步

这是最容易引起回归的部分。

应对：

- 先做 adapter 层，不立即动 viewer
- 使用 feature flag 或兼容分支渐进切换
- 保留 fallback 到旧 replay 方式的临时开关

### C. workflow/sync 契约受跨仓影响

`workflow/sync` 聚合字段来自后端仓，改动不完全由本仓控制。

应对：

- 前端先做 adapter 兼容多字段
- 文档中明确“前端兼容期”“后端目标期”两套口径

## 8.2 中风险项

- 评论实时同步导致重复插入
- task scope 变更后影响现有草稿恢复
- 测量与批注统一回放逻辑被拆分后出现遗漏

## 9. 验收标准

## 9.1 功能验收

### 批注与评论

- 可以创建/编辑/删除批注
- 可以新增/编辑/删除评论
- 评论不因确认记录重放而丢失
- 评论可跨嵌入恢复正确挂回对应批注

### 确认记录

- 批注与测量可以确认并持久化
- 已确认数据与草稿数据分层展示
- 同任务可多次确认，互不覆盖

### 恢复链

- task 内恢复与 workflow/sync 恢复结果一致
- 导入导出后仍能正确重建批注、评论、附件与测量

### 协同

- 多人评论可以实时同步
- 任务切换不会串扰草稿

## 9.2 代码验收

- adapter / service / domain 分层清晰
- `ReviewPanel.vue` 不再承担过多数据归一职责
- `reviewRecordReplay.ts` 职责明显收敛
- `useToolStore.ts` 不再是评论的唯一真源

## 9.3 文档验收

- 字段与接口迁移文档更新
- 前后端协作项单独列清单
- 验证文档补充多任务、多人协同、嵌入恢复场景

## 10. 推荐执行顺序

建议按以下顺序执行：

1. **Phase A**：冻结现状与字段口径
2. **Phase B**：统一 ReviewSnapshot 中间层
3. **Phase C**：评论从 annotation 内联结构解耦
4. **Phase F**：评论实时同步统一化
5. **Phase D**：引入 annotationKey / workflowNode / reviewRound
6. **Phase G**：task 级草稿隔离
7. **Phase E**：draft / confirmed 双层渲染
8. **Phase H**：事件流与差异审计增强

说明：

- `Phase E` 技术风险最高，建议放在前面基础模型稳定后再做。
- `Phase B + C + F` 是当前收益最大、风险相对可控的主线。

## 11. 结论

当前批注体系的问题不是“功能缺失”，而是“结构尚未为多人协同、跨系统同步和长期产品化准备好”。  
本计划的核心思想不是推翻现有实现，而是在现有可用链路上，逐步引入：

- 统一 snapshot 中间层
- 评论独立真源
- 稳定批注业务键
- draft / confirmed 双层模型
- task 级草稿隔离

这样可以在不打断现有交付节奏的前提下，把三维校审批注体系演进为一套真正可扩展、可协同、可审计的正式架构。
