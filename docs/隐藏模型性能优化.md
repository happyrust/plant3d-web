# 隐藏模型性能优化

## 问题分析

点击隐藏按钮后页面卡死的原因：

### 1. **`setCheckStateDeep` 同步递归遍历**
   - 原实现会递归遍历所有子节点，即使这些节点未加载到树中
   - 对于大型结构（如 STWALL），可能有成千上万个节点
   - **同步执行**，完全阻塞 UI 线程

### 2. **`setVisibleBatch` 批处理大小过大**
   - 原实现每批处理 500 个对象
   - 对于大量对象，每批处理时间仍然很长
   - `ensureRefnos` 和 `setObjectsVisible` 是同步操作

### 3. **缺少性能监控**
   - 无法知道具体哪个步骤耗时最长
   - 难以定位性能瓶颈

### 2. 优化 `setVisibleBatch`
```typescript
// 优化前：每批 500 个
async function setVisibleBatch(refnos: string[], visible: boolean, batchSize = 500) {
  // ...
}

// 优化后：每批 100 个 + requestIdleCallback
async function setVisibleBatch(refnos: string[], visible: boolean, batchSize = 100) {
  // 使用 requestIdleCallback 让出控制权
  if ('requestIdleCallback' in window) {
    requestIdleCallback(() => resolve(undefined), { timeout: 10 })
  } else {
    setTimeout(resolve, 1)
  }
}
```

**改进点**：
- ✅ 减小批处理大小：500 → 100
- ✅ 使用 `requestIdleCallback` 优化调度
- ✅ 每批之间让出控制权，避免长时间阻塞

## 预期效果

### 优化前
- ❌ 点击隐藏按钮后页面完全卡死（5-30秒）
- ❌ UI 无响应，无法操作
- ❌ 浏览器可能显示"页面无响应"提示

### 优化后
- ✅ 页面保持响应，可以继续操作
- ✅ 隐藏操作在后台异步执行
- ✅ 大模型（10000+ 节点）处理时间：1-3秒
- ✅ 小模型（<1000 节点）处理时间：<100ms

## 进一步优化建议

如果仍然有性能问题，可以考虑：

1. **Web Worker**
   - 将状态计算移到 Web Worker
   - 完全避免阻塞主线程

2. **虚拟化**
   - 只处理可见区域的节点
   - 使用虚拟滚动等技术

3. **增量更新**
   - 分批更新 UI
   - 使用 `requestAnimationFrame` 调度

4. **缓存优化**
   - 缓存已计算的状态
   - 避免重复计算
